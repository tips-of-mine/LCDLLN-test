# M100.46 — Zone Presets Library (Phase 12 — Accessibilité)

## Résumé

Catalogue de **templates de zones jouables prêts à l'emploi**. Chaque preset est un fichier JSON qui décrit une séquence ordonnée de commandes éditeur (M100.35-43) à exécuter sur une zone vide pour produire un terrain cohérent : « Forêt tempérée », « Côte rocheuse », « Désert », « Plateau enneigé », etc.

Le mappeur lance le dialog **« Nouvelle zone depuis preset »** depuis le menu `Fichier`, choisit un template parmi 8 thumbnails, ajuste 3 curseurs globaux (relief, eau, sécheresse) et un seed, puis clique « Créer la zone ». L'éditeur exécute alors les commandes une par une avec une modale de progression qui affiche chaque étape (« Pose de la chaîne de montagnes… », « Génération du réseau hydrographique… »). En 30 secondes à 2 minutes selon la complexité, le mappeur obtient une zone jouable de base qu'il peut ensuite enrichir manuellement.

Chaque preset référence des **presets d'outils** (M100.45) plutôt que des paramètres bruts, ce qui garde le JSON lisible et évolutif : si un preset d'outil change ses valeurs par défaut, les zone presets en bénéficient automatiquement.

**Une section `decoration: []` est laissée vide pour le MVP**, structurée pour accueillir la Phase 13 (arbres, champs, fleurs, structures linéaires, panneaux, cailloux) sans refactor du format. Ce ticket annonce explicitement que les zones produites sont visuellement « nues » côté végétation, et que Phase 13 viendra les enrichir.

**Aucun nouvel exécutable.** Code dans `engine_core`.

## Dépendances

- **M100.1** — World Editor Bootstrap.
- **M100.2** — Command Stack & Undo/Redo (les commandes du preset y sont poussées).
- **M100.35-39** — Tous les outils de terrain (consommés par les presets).

## Contexte critique à respecter

### 1. La décoration sera ajoutée par Phase 13 (sans refactor)

Le format JSON inclut dès maintenant un champ `decoration: []` qui sera enrichi par Phase 13. **Aucune dépendance dure** sur les outils de Phase 13 dans ce ticket — la section reste vide, ignorée si vide, l'éditeur ne lève aucune erreur. Quand Phase 13 livrera Vegetation Painter, Linear Structures, etc., ce ticket sera revisité (M100.46b ?) pour peupler les 8 presets avec de la décoration adéquate. **Le format JSON ne changera pas**.

### 2. Reproductibilité par seed

Un preset + un seed donné doit produire **exactement la même zone** à chaque exécution (byte-equal sur `terrain.bin`). Cela impose :
- Toutes les commandes du preset utilisent le seed comme RNG seed lorsqu'applicable (érosion hydraulique, distribution de sources, etc.).
- Les positions géographiques des éléments macro (chaînes de montagnes, lignes de côte) sont **explicites** dans le JSON, pas générées aléatoirement. Le seed n'introduit que des micro-variations.
- L'ordre des opérations est déterministe.

### 3. Customisation par 3 curseurs globaux

Au moment de l'exécution, le mappeur peut tweaker 3 sliders qui **scalent uniformément** certains paramètres du preset :

| Slider | Effet |
|---|---|
| **Niveau de relief** | Multiplie les hauteurs des montagnes + amplitude du sculpt macro. 0.0 = plat absolu, 1.0 = défaut preset, 2.0 = double altitude. |
| **Densité de l'eau** | Multiplie le nombre de rivières du watershed et le nombre de gouttes hydraulic. 0.0 = pas d'eau, 1.0 = défaut, 2.0 = très humide. |
| **Sécheresse** | Augmente l'évaporation hydraulic + diminue la densité de rivières + favorise le wind erosion. 0.0 = humide, 1.0 = défaut, 2.0 = aride. |

Le preset JSON marque explicitement quels paramètres sont **sensibles** à chaque curseur via des tags `affectedBy: ["relief"]`. Les autres paramètres sont insensibles aux curseurs.

### 4. Exécution sur zone vide UNIQUEMENT

Les zone presets s'appliquent à une **zone nouvellement créée** (terrain plat, splat par défaut, pas de water surfaces, pas de mesh inserts). Si le mappeur veut appliquer un preset à une zone existante, on lui dit clairement : « Cela écrasera votre zone actuelle. Voulez-vous continuer ? » → si oui, reset complet puis exécution.

Pas de fusion partielle, pas de « appliquer seulement la partie érosion ». C'est tout-ou-rien.

### 5. Les commandes du preset sont Undoable individuellement

Chaque commande exécutée par le preset est **poussée sur le `CommandStack`** comme une commande utilisateur normale. Le mappeur peut donc :
- Annuler la dernière étape (Ctrl+Z) si elle ne lui plaît pas.
- Voir l'historique étape par étape dans le panel History.
- Annuler complètement l'application du preset (N undos d'affilée).

Conséquence importante : le preset n'introduit pas de « super-commande » macro. Chaque opération reste atomique.

### 6. Invariant terrain visible

L'application du preset utilise les outils existants qui respectent déjà l'invariant. Aucun changement render side.

### 7. Une seule application

Pas de nouvel exécutable hors la suite `zone_presets_tests`.

### 8. Arbo server-core-style

- `src/world_editor/zone_presets/` (nouveau).
- `game/data/editor/zone_presets/*.json` (committés).
- `game/data/editor/zone_presets/thumbnails/*.png` (committés, 256×144).

##### A.2 — Format d'une opération

Chaque entrée du tableau `operations` est typée par `type` (qui correspond au `toolId` de M100.45) :

```json
{
  "type": "mountain_macro",
  "preset": "pre_alpine",
  "polyline": [[1000, 4000], [3000, 4000], [5000, 4000]],
  "widthMeters": 800,
  "heightMeters": 600,
  "affectedBy": ["relief"]
}
```

Le moteur d'exécution :
1. Cherche le `toolId == "mountain_macro"` dans la registry.
2. Charge le preset `"pre_alpine"` depuis `tool_presets/mountain_macro.json` (M100.45).
3. Override les paramètres explicites du JSON preset zone (ici `polyline`, `widthMeters`, `heightMeters`).
4. Si `affectedBy` contient `"relief"` : multiplie les hauteurs par le curseur global Relief.
5. Crée la `ICommand` correspondante et l'exécute.

##### C.3 — Workflow

1. Mappeur clique « Nouvelle zone depuis preset… ».
2. Dialog s'ouvre, grille de 8 thumbnails.
3. Sélection d'un preset → description + estimated time s'affichent.
4. Customisation (3 sliders + seed) optionnelle.
5. Click « Créer la zone » → confirmation si la zone actuelle n'est pas vide.
6. Modale de progression apparaît, exécution :

│  Temps restant estimé : 33 s                 │

##### C.4 — Annulation pendant exécution

Le bouton « Annuler » dans la modale arrête l'exécution **après la commande en cours** (pas en plein milieu d'une simulation hydraulique). Les commandes déjà exécutées restent sur le CommandStack (peuvent être annulées normalement via Ctrl+Z).

#### Partie D — Moteur d'exécution

##### D.1 — `ZonePresetExecutor`

```cpp
namespace engine::editor::world::zone_presets
{
    struct CustomizationParams
    {
        float reliefMultiplier   = 1.0f;
        float waterDensityMultiplier = 1.0f;
        float drynessMultiplier  = 1.0f;
        uint32_t seed            = 42;
    };

    struct ExecutionProgress
    {
        uint32_t currentStep;
        uint32_t totalSteps;
        std::string currentStepLabel;  // "Pose de la chaîne de montagnes…"
        float fractionComplete;        // 0.0 .. 1.0
        bool isCancelled;
    };

    class ZonePresetExecutor
    {
    public:
        /// Exécute le preset sur la zone actuelle (reset implicite si non vide,
        /// après confirmation utilisateur).
        /// progressCallback est invoqué synchrone à chaque pas.
        /// Renvoie true si terminé, false si annulé par l'user.
        bool Execute(const ZonePreset& preset,
                     const CustomizationParams& params,
                     std::function<bool(const ExecutionProgress&)> progressCallback);

        /// Signale à l'exécution en cours qu'elle doit s'arrêter après la
        /// commande courante. Appelable depuis le thread UI.
        void RequestCancel();
    };
}
```

##### D.2 — Application des multiplicateurs `affectedBy`

Pour chaque opération avec `affectedBy: ["relief"]` :
- Champs concernés : `heightMeters`, `amplitudeMeters`, `depthMeters` (selon le type d'op).
- Valeur effective = valeur JSON × `reliefMultiplier`.

Pour `["water_density"]` :
- Champs concernés : `numDroplets`, nombre de sources de rivière, `flowAccumThreshold`.

Pour `["dryness"]` :
- Champs concernés : `evaporationRate` (+), `numRivers` (-), `windStrength` (+).

Les opérations sans `affectedBy` utilisent leurs valeurs JSON telles quelles.

##### D.3 — Reset de la zone

Si la zone actuelle n'est pas vide :
- Dialog de confirmation.
- Si confirmé : `WorldMapEditDocument::Reset()` qui :
  - Vide `TerrainDocument` (heightmap à 0, splat par défaut).
  - Vide `WaterDocument`.
  - Vide `MeshInsertDocument`.
  - Vide `InstanceDocument`.
  - Push une `ResetZoneCommand` sur le CommandStack (réversible par Ctrl+Z dans la limite de la mémoire).

##### D.4 — Reproductibilité

Le `seed` est passé comme `rngSeed` à toutes les opérations qui en supportent un (`hydraulic_erosion`, `thermal_wind_erosion` particle, distribution de rivières si non-déterministe). Garantit que `Execute(preset, params)` est byte-equal pour un même `(preset, params)`.

#### Partie E — Tests

`zone_presets_tests` :
- Round-trip JSON (write/read) pour les 8 presets MVP.
- Validation : chaque preset référence des `toolId` et `tool preset` existants.
- Exécution déterministe : `Execute(preset, params)` deux fois → terrain.bin byte-equal.
- Customisation : `reliefMultiplier = 2.0` double effectivement les hauteurs.
- Annulation : commande courante terminée, commandes suivantes ignorées.
- Section `decoration: []` ignorée silencieusement.
- Section `decoration` avec types inconnus → warning log, pas crash.

### Structures (extraits)

`ZonePreset.h` :

```cpp
namespace engine::editor::world::zone_presets
{
    struct LocalizedString
    {
        std::string fr;
        std::string en;
        // ... extensible : de, es, etc.
        const std::string& Get(const std::string& lang) const;
    };

    struct ZonePresetOperation
    {
        std::string type;                       // "mountain_macro", "hydraulic_erosion", ...
        std::string toolPresetId;               // optionnel, "realistic", etc.
        nlohmann::json parameters;              // brut, dispatch via OperationDispatcher
        std::vector<std::string> affectedBy;    // ["relief", "water_density", "dryness"]
    };

    struct ZonePreset
    {
        int version = 1;
        std::string id;
        LocalizedString displayName;
        LocalizedString description;
        std::string thumbnailPath;

  → Si confirmé : WorldMapEditDocument.Reset()
  → ZoneCreationProgressModal s'affiche

[Exécution séquentielle des opérations]
  → Étape 1/8 : Mountain macro (12 s)
    → MountainCommand pushed → executed
  → Étape 2/8 : Valley macro (3 s)
  → Étape 3/8 : Coastline (8 s)
  → ... etc.

[Si user click Annuler]
  → RequestCancel(), commande courante finit, suivantes ignorées
  → Modale se ferme, zone partielle dans le viewport

[Si exécution complète]
  → Modale se ferme
  → Viewport montre la zone, History panel liste les 8 commandes
  → User peut Ctrl+Z pour annuler la dernière étape, ou Ctrl+Z répété pour tout retirer
```
