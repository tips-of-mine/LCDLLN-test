## Ticket content (M02.3b) — référence

# M02.3b — FrameGraph : migration vers VK_KHR_synchronization2 (Vulkan 1.3)

## Objectif
Remplacer les appels `vkCmdPipelineBarrier` (Vulkan 1.0) dans `FrameGraph.cpp` par
`vkCmdPipelineBarrier2` (Vulkan 1.3 / `VK_KHR_synchronization2`). L'API sync2 offre
des `VkPipelineStageFlags2` plus granulaires, élimine les over-synchronizations
actuelles sur les passes compute (AutoExposure M08.3) et prépare le moteur aux
besoins de synchronisation plus précis des passes futures.

## Contexte
`FrameGraph::emitBarriersBeforePass` (M02.3) utilise `vkCmdPipelineBarrier` avec
`VkPipelineStageFlags` 32 bits. Plusieurs stages sont trop larges :
- `VK_PIPELINE_STAGE_FRAGMENT_SHADER_BIT` couvre l'intégralité des shaders fragment
  alors que seul l'accès sampled est requis dans la majorité des passes.
- Les passes compute d'AutoExposure (M08.3) ne distinguent pas lecture/écriture
  storage buffer avec les flags legacy.

Avec sync2, `VK_ACCESS_2_SHADER_SAMPLED_READ_BIT` est séparé de
`VK_ACCESS_2_SHADER_STORAGE_READ_BIT`, permettant des barrières plus précises
sans modifier le comportement observable.

## Dépendances
- M02.3 (FrameGraph barrières existantes)
- M08.3 (AutoExposure compute — principale bénéficiaire)
- Vulkan 1.3 ou extension `VK_KHR_synchronization2` (à vérifier dans VkDeviceContext)

## Livrables
- `engine/render/VkDeviceContext.h/.cpp`
  - Activer l'extension `VK_KHR_synchronization2` lors de la création du device si Vulkan < 1.3.
  - Stocker `bool m_sync2Supported`.
  - Exposer `bool SupportsSynchronization2() const`.
- `engine/render/FrameGraph.h`
  - `ResourceUsageState` : remplacer `VkPipelineStageFlags` → `VkPipelineStageFlags2`,
    `VkAccessFlags` → `VkAccessFlags2`.
- `engine/render/FrameGraph.cpp`
  - `GetUsageState` : retourner les flags sync2 (voir tableau ci-dessous).
  - `emitBarriersBeforePass` : émettre `VkImageMemoryBarrier2` + `VkDependencyInfo` + `vkCmdPipelineBarrier2`.
  - Conserver un fallback `vkCmdPipelineBarrier` (Vulkan 1.0) si sync2 non disponible.

## Spécification technique
### Mapping ImageUsage → sync2 stages + access
| ImageUsage | VkPipelineStageFlags2 | VkAccessFlags2 |
| ColorWrite | COLOR_ATTACHMENT_OUTPUT | COLOR_ATTACHMENT_WRITE |
| DepthWrite | EARLY_FRAGMENT_TESTS \| LATE_FRAGMENT_TESTS | DEPTH_STENCIL_ATTACHMENT_WRITE |
| SampledRead | FRAGMENT_SHADER \| COMPUTE_SHADER | SHADER_SAMPLED_READ |
| TransferSrc | TRANSFER | TRANSFER_READ |
| TransferDst | TRANSFER | TRANSFER_WRITE |

### Barrière sync2
VkImageMemoryBarrier2 + VkDependencyInfo + vkCmdPipelineBarrier2.

### Détection runtime
VkPhysicalDeviceSynchronization2FeaturesKHR dans pNext de VkDeviceCreateInfo si Vulkan < 1.3.

## Étapes d'implémentation
1. Vérifier et activer `synchronization2` dans `VkDeviceContext` (feature 1.3 ou extension KHR).
2. Mettre à jour `ResourceUsageState` avec `Flags2`.
3. Mettre à jour `GetUsageState` avec le tableau sync2.
4. Réécrire `emitBarriersBeforePass` : path sync2 + fallback legacy.
5. Valider avec Vulkan Validation Layers en mode `best_practices`.

## Notes / pièges à éviter
- `vkCmdPipelineBarrier2` est le nom Vulkan 1.3 natif; en Vulkan 1.2 avec extension, c'est `vkCmdPipelineBarrier2KHR`. Utiliser un alias selon la version détectée.
- Le fallback legacy est nécessaire pour les GPU Intel HD Vulkan 1.1 — ne pas le supprimer.
- Ne pas migrer les barrières internes à `ShadowMapPass` ou `GeometryPass` dans ce ticket : seul le FrameGraph est dans la portée.
