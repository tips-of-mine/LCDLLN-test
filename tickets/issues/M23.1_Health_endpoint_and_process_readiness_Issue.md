# M23.1 — Health endpoint (/healthz) + readiness

**Status:** Closed

## Objectif
Exposer un endpoint health/readiness pour Master et Shard afin de faciliter supervision et orchestration.

## Dépendances
- M22 (services)

## Livrables
- HTTP endpoint minimal:
  - `/healthz`: 200 OK si process alive
  - `/readyz`: 200 OK si prêt (DB OK, migrations OK, shards registry OK)
- Codes d'état et payload JSON minimal

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Port séparé du port jeu.
- Option config pour bind address (localhost par défaut en dev).

## Étapes d'implémentation
1. Implémenter mini HTTP server (lib légère) ou endpoint interne.
2. Définir checks readiness (DB ping, migration done).
3. Ajouter logs sur transitions not-ready -> ready.

## Critères de validation (DoD)
- [x] healthz répond toujours.
- [x] readyz reflète DB down/up.

### 2) COMMANDES WINDOWS / LINUX À EXÉCUTER

Le health endpoint est **UNIX only** (Master et Shard Linux). Build Linux :

- `cmake --preset linux-x64-release` (ou `linux-x64`)
- `cmake --build --preset linux-x64-release`

Exécutables :
- Master : `build/linux-x64-release/pkg/server/lcdlln_server` — endpoint sur `server.health.port` (défaut 3842), bind `server.health.bind` (défaut 127.0.0.1).
- Shard : `build/linux-x64-release/pkg/server/lcdlln_shard` — endpoint sur `server.health.port` (défaut 3843 pour éviter conflit avec Master).

Test manuel :
- `curl http://127.0.0.1:3842/healthz` → 200 `{"status":"ok"}`
- `curl http://127.0.0.1:3842/readyz` → 200 si DB OK, 503 `{"status":"not_ready","reason":"db"}` si DB down.
