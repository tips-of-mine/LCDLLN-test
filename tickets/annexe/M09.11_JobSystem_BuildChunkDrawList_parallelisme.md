# M09.11 — Job System : parallélisation de BuildChunkDrawList

## Objectif
Intégrer le Job System (M00.3) dans `BuildChunkDrawList` pour traiter les chunks
en parallèle. Le Job System est entièrement implémenté et validé depuis M00.3 mais
n'est utilisé dans aucun système du moteur à ce jour (M00–M09).

## Contexte
`BuildChunkDrawList` (M09.5) itère séquentiellement sur l'ensemble des chunks requis :
frustum test + distance + décision HLOD pour chacun. Avec les rings Active (5×5 = 25),
Visible (7×7 = 49) et Far (11×11 = 121) actifs simultanément, c'est jusqu'à ~195 chunks
traités en boucle mono-thread à chaque frame. Ce ticket active enfin le Job System
sur ce cas d'usage concret.

Ce ticket dépend de M09.7 (GlobalChunkCoord) pour la cohérence des types.

## Dépendances
- M00.3 (Job System : `Jobs::Enqueue`, `Jobs::Wait`, `JobGroup`)
- M09.5 (BuildChunkDrawList existant)
- M09.7 (GlobalChunkCoord — doit être livré avant ce ticket)
- M09.8 (optimisation distSq — recommandé avant pour le hot path job)

## Livrables
- `engine/world/HlodRuntime.h`
  - Extraire `ChunkDrawDecision ComputeChunkDecision(const ChunkRequest&, Vec3Ref, FrustumRef, const HlodRuntime&, float)` comme fonction libre (ou méthode statique).
- `engine/world/HlodRuntime.cpp`
  - Implémenter `ComputeChunkDecision`.
  - Dans `BuildChunkDrawList` :
    - Si `count < kChunkJobThreshold` (config, défaut 64) → path séquentiel existant.
    - Sinon → découper en tranches de `kChunkSliceSize` (défaut 64) et dispatcher via `Jobs::Enqueue` + `Jobs::Wait`.
  - Construire la debug string par agrégation post-wait.
- `engine/Engine.cpp` : aucun changement requis (interface `BuildChunkDrawList` inchangée).

## Spécification technique
```cpp
constexpr size_t kChunkJobThreshold = 64; // en dessous : path séquentiel (overhead > gain)
constexpr size_t kChunkSliceSize    = 64; // chunks par job

// Pré-allouer avant dispatch (pas d'accès concurrent au vecteur lui-même)
decisionsOut.resize(count);

engine::core::jobs::JobGroup group;
const size_t sliceCount = (count + kChunkSliceSize - 1) / kChunkSliceSize;

for (size_t s = 0; s < sliceCount; ++s)
{
    const size_t begin = s * kChunkSliceSize;
    const size_t end   = std::min(begin + kChunkSliceSize, count);
    Jobs::Enqueue(group, [&, begin, end]() {
        for (size_t i = begin; i < end; ++i)
            decisionsOut[i] = ComputeChunkDecision(
                requestedChunks[i], cameraPosition, frustum, hlod, maxDrawDistanceMeters);
    });
}
Jobs::Wait(group);

return AggregateDebugString(decisionsOut); // cumul hlod/inst/culled
```
- Chaque job écrit dans sa propre plage d'indices → pas de mutex.
- `cameraPosition`, `frustum`, `hlod` sont en lecture seule pendant le dispatch.
- Seuil `kChunkJobThreshold` lisible depuis config `world.chunk_job_threshold`.

## Étapes d'implémentation
1. Extraire `ComputeChunkDecision` depuis la boucle existante de `BuildChunkDrawList`.
2. Pré-allouer `decisionsOut.resize(count)`.
3. Ajouter la branche jobs (tranches + dispatch + wait).
4. Conserver le path séquentiel pour `count < threshold`.
5. Implémenter `AggregateDebugString` pour la debug string finale.
6. Valider que les résultats sont identiques entre les deux paths.

## Critères de validation (DoD)
- [ ] Build Windows OK (configure + build) via presets existants.
- [ ] Exécutable lancé sans crash.
- [ ] Pour N = 200 chunks : résultats identiques au path séquentiel (`useHlod`, `culled` identiques chunk par chunk).
- [ ] Pour N = 10 chunks (< threshold) : path séquentiel utilisé (aucun job enqueué).
- [ ] Aucune data race (tester avec plusieurs threads actifs dans le job system).
- [ ] Aucun dossier racine non autorisé créé.
- [ ] Rapport final : fichiers modifiés + commandes + résultats + DoD.

## Notes / pièges à éviter
- Ne pas appeler `decisionsOut.push_back` depuis les jobs : utiliser `resize` + accès indexé.
- `Jobs::Wait` depuis le thread principal est safe (M00.3 pompe les jobs en attente pour éviter le deadlock).
- Ne pas créer un job par chunk individuel (overhead de scheduling supérieur au gain).
- Les compteurs hlod/inst/culled doivent être agrégés après le wait, pas accumulés avec des atomiques.
