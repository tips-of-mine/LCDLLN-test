# M20.5 — Auth/Register flows: opcodes + handlers (Master)

**Status:** Closed

## Objectif
Implémenter les opcodes v1 pour inscription et authentification (sur Master), et le flux complet client↔serveur.

## Dépendances
- M19.1 (opcodes spec)
- M19.5/M19.6 (TLS)
- M20.2 (argon2)
- M20.3 (sessions)
- M21 (DB)

## Livrables
- Opcodes:
  - `REGISTER_REQUEST/RESPONSE`
  - `AUTH_REQUEST/RESPONSE`
- Handlers serveur (Master):
  - register: create account (sans personnages)
  - auth: verify + create session
- Client:
  - UI/flow minimal (headless acceptable) pour register/auth via TLS
- Error mapping via ERROR packet si nécessaire

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Register:
  - validate input
  - check existing email/login
  - store final_hash + server_salt
- Auth:
  - validate input
  - fetch account, check status active
  - verify final_hash
  - enforce single-session
  - return session_id + server time + version gating hooks

## Étapes d'implémentation
1. Définir payload binaire request/response (doc alignée).
2. Implémenter handlers serveur.
3. Implémenter client request/response via dispatcher.
4. Tests intégration: register then auth.

## Critères de validation (DoD)
- [ ] Register crée compte sans personnage.
- [ ] Auth retourne session_id.
- [ ] Single-session enforce.
