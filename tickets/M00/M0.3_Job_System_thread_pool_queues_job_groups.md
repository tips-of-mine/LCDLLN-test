# M0.3 — Job System: thread pool, queues, job groups

## Objectif
Mettre en place un système de jobs pour streaming/asset build/render prep : thread pool, work stealing simple, groupes + wait.

## Dépendances
- M0.1
- M0.2

## Livrables
- `/engine/core/jobs/JobSystem.*`
- `JobHandle`/`JobGroup` (Wait/IsDone)
- Queues thread-safe (MPSC) + worker threads
- Tests: 100k jobs + nested jobs


## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

## Spécification technique
- API: `Jobs::Init(n)`, `Jobs::Enqueue(fn)`, `Jobs::Wait(handle/group)`.
- Support des jobs courts; éviter allocations par job (pool/arena).
- Possibilité d'annuler via token (option MVP) ou versioning au-dessus.

## Étapes d’implémentation
1. Créer thread pool (n = cores-1) + stop flag.
2. Implémenter queue thread-safe (mutex/condvar MVP).
3. Définir JobHandle (counter) et JobGroup (atomic count).
4. Ajouter `Wait` qui pump aussi des jobs (avoid deadlock).
5. Écrire tests (1k, 100k, nested, multi-producer).

## Critères de validation (DoD)
- [ ] 100k jobs exécutés < quelques secondes (selon machine) sans deadlock.
- [ ] Wait fonctionne même si appelé depuis worker.
- [ ] Shutdown propre (join threads).

## Notes / pièges à éviter
- MVP mutex ok; optimise lock-free plus tard.
- Éviter jobs trop gros (préférer découpage).
