# M100.49 — Tutoriel interactif & Diagnostic (Phase 12 — Accessibilité)

## Résumé

Deuxième volet du système d'aide (après M100.47 Tooltips & F1 Doc), avec deux mécaniques complémentaires :

1. **Tutoriel interactif au premier lancement** : guidage en 8 étapes où le mappeur exécute lui-même les actions (avec surlignage du widget à cliquer, instructions overlay, validation de l'action). Couvre le parcours : lancer un zone preset → modifier un paramètre simple → valider → exporter. À la fin, l'utilisateur a produit une zone jouable, l'a validée et l'a exportée. Le tutoriel est skippable et reprenable depuis le menu `Aide`.

2. **Système de diagnostic « Pourquoi ça ne marche pas ? »** : bouton accessible depuis n'importe où dans l'éditeur qui analyse l'état courant et propose des suggestions contextualisées. Exemples : « Tes rivières flottent au-dessus du terrain → tu as érodé après avoir posé les rivières. [Re-lancer le watershed] [Comprendre pourquoi] » ; « Aucun outil ne semble actif → as-tu cliqué sur un outil dans la toolbar ? ». Le diagnostic s'appuie sur le `ZoneValidator` (M100.48) augmenté de **règles « contextuelles »** qui détectent des incohérences d'usage typiques, et propose des actions correctives en un clic quand possible.

Ces deux mécaniques utilisent une infrastructure commune (`OverlayGuidanceSystem`) qui sait surligner des widgets, afficher des bulles d'instruction, et chaîner des étapes.

**Aucun nouvel exécutable.** Code dans `engine_core`.

## Dépendances

- **M100.45** — Simple/Advanced Mode (le tutoriel utilise le mode Simple par défaut).
- **M100.46** — Zone Presets Library (le tutoriel utilise « Forêt tempérée » comme exemple).
- **M100.47** — Tooltips & F1 Doc (le tutoriel peut référencer des sections de doc).
- **M100.48** — Zone Validation Service (le diagnostic réutilise le moteur de règles).

## Contexte critique à respecter

### 1. Le tutoriel exécute les actions vraies, pas un mode démo

Quand l'utilisateur clique « Créer la zone » dans l'étape 3 du tutoriel, **la zone est vraiment créée**. Pas de mode simulation. À la fin du tutoriel, le mappeur a entre les mains une vraie zone valide qu'il peut continuer à éditer ou exporter. C'est important pour l'apprentissage par la pratique.

### 2. Skippable et reprenable

À tout moment, l'utilisateur peut :
- Cliquer **« Passer le tutoriel »** → tutoriel marqué comme passé, ne se relancera pas auto.
- Cliquer **« Quitter pour l'instant »** → progression sauvée, possibilité de reprendre depuis `Aide → Reprendre le tutoriel`.
- Cliquer **« Recommencer le tutoriel »** depuis `Aide` à tout moment.

Persistance via `UserPrefsStore.tutorialCompletionFlags` (M100.45).

### 3. Le diagnostic propose, ne décide pas

Le bouton « Pourquoi ça ne marche pas ? » liste des suggestions, **mais n'exécute jamais d'action sans confirmation explicite**. Les boutons « one-click fix » demandent toujours une confirmation modale « Voulez-vous re-lancer le watershed maintenant ? [Oui] [Non, je préfère le faire moi-même] ».

### 4. Diagnostic différent de validation

| | Validation (M100.48) | Diagnostic (ce ticket) |
|---|---|---|
| Quand ? | Sur demande / avant export | Sur demande, focus sur « ça marche pas » |
| Quoi ? | Cohérence donnée | Cohérence d'usage / workflow |
| Sorties | Issues à corriger | Suggestions actionnables |
| Cible | Validité technique | Compréhension du flow |

Exemple :
- Validation : « ❌ Donjon avec ID dupliqué » (problème de données)
- Diagnostic : « Tu as activé Hydraulic mais le bouton Simulate ne réagit pas. Tu as 0 chunks chargés. Crée d'abord une zone (Fichier → Nouvelle zone depuis preset). » (problème de workflow)

Les deux systèmes partagent l'architecture des règles (`IDiagnosticRule` hérite des concepts de `IValidationRule`) mais sont accessibles via des UIs distinctes.

### 5. Pas d'IA générative

Toutes les règles de diagnostic et toutes les étapes du tutoriel sont **codées en dur** ou définies en JSON. Pas de modèle LLM appelé en ligne. Le contenu est 100 % offline et déterministe.

### 6. Invariant terrain visible

Surlignage des widgets via `ImGui::GetForegroundDrawList()` (rectangles colorés autour des widgets cibles). Bulles d'instruction = panels ImGui standards. Aucun render pass dédié, aucune touche caméra.

### 7. Une seule application

Pas de nouvel exécutable hors la suite `tutorial_diagnostic_tests`.

### 8. Arbo cmangos-style

- `src/world_editor/tutorial/` (nouveau).
- `src/world_editor/diagnostic/` (nouveau).
- `src/world_editor/help/` (étendu).
- `game/data/editor/tutorials/*.json` (committés).
- `game/data/editor/diagnostic_rules/*.json` (committés).

### Inclus

#### Partie A — `OverlayGuidanceSystem` infrastructure commune

Service partagé entre tutoriel et diagnostic pour surligner des widgets et afficher des instructions.

```cpp
namespace engine::editor::world::help
{
    /// ID stable d'un widget cible dans l'UI (e.g. "menubar.file.new_from_preset",
    /// "toolbar.button.hydraulic_erosion", "panel.tool_properties.numDroplets_slider").
    /// Convention : <panel>.<sous-élément>.<id>
    using WidgetTargetId = std::string;

    struct OverlayInstruction
    {
        WidgetTargetId      targetWidget;   // widget à surligner (vide = pas de surlignage)
        std::string         titleFr;
        std::string         bodyFr;
        std::string         iconHint;       // "👆", "✓", "⚠", ...
        bool                requiresAction; // si true, on attend que l'user fasse l'action
        WidgetTargetId      validatesOnAction; // signale au système que l'action a été faite
        std::function<bool()> customValidator;  // optionnel, validation custom
    };

    /// Système central qui :
    /// 1. Reçoit l'enregistrement des widgets par leur ID.
    /// 2. Affiche l'overlay (surlignage + bulle) sur le widget courant.
    /// 3. Détecte quand l'utilisateur a fait l'action attendue.
    /// 4. Permet d'avancer à l'étape suivante.
    class OverlayGuidanceSystem
    {
    public:
        static OverlayGuidanceSystem& Instance();

        /// Appelée par chaque widget de l'UI au moment de son render
        /// pour s'enregistrer comme cible potentielle.
        void RegisterWidget(const WidgetTargetId& id, ImVec2 topLeft, ImVec2 bottomRight);

        /// Lance une séquence d'instructions. Si une instruction est déjà
        /// active, elle est remplacée.
        void StartSequence(const std::vector<OverlayInstruction>& instructions,
                            std::function<void()> onComplete = nullptr);

        /// Demande d'avancer à l'étape suivante.
        void AdvanceStep();

        /// Quitte la séquence en cours.
        void AbortSequence();

        /// True si une séquence est active (cache certains UI éléments).
        bool IsActiveSequence() const;

        /// Render pendant le frame UI (à appeler depuis WorldEditorShell::OnRenderUI).
        void RenderOverlay();
    };
}
```

##### A.1 — Surlignage visuel

Quand `IsActiveSequence()` retourne true, le système :
1. Tint l'écran d'un voile semi-transparent (40 % noir) sauf autour du widget cible.
2. Trace un rectangle pulsant (alpha animé) autour du widget cible.
3. Affiche une bulle d'instruction proche du widget (positionnement intelligent pour ne pas cacher le widget).

##### A.2 — Détection d'action

Plusieurs façons de détecter qu'une action a été faite :
- `validatesOnAction = "<sameId>"` : action sur ce widget (clic, change de valeur).
- `customValidator` : lambda qui inspecte un état (`return zonePresetDialog.IsOpen();`).
- Pour l'étape « Crée une zone » : detection via `WorldMapEditDocument.IsZoneNonEmpty()`.

##### C.2 — `IDiagnosticRule` interface

Similaire à `IValidationRule` mais avec des sorties différentes :

```cpp
namespace engine::editor::world::diagnostic
{
    enum class SuggestionImportance : uint8_t
    {
        Tip    = 0,
        Strong = 1,  // forte suspicion que c'est ça
        Critical = 2, // bloquant ou cause certaine
    };

    struct DiagnosticSuggestion
    {
        std::string ruleId;
        std::string titleFr;
        std::string explanationFr;
        SuggestionImportance importance;
        std::string docSectionId;       // lien M100.47

        // Action one-click optionnelle
        std::string oneClickActionLabelFr;  // ex: "Re-lancer le watershed"
        std::function<void()> oneClickAction;
        std::string confirmationMessageFr;  // pour la modal de confirmation
    };

    class IDiagnosticRule
    {
    public:
        virtual ~IDiagnosticRule() = default;
        virtual const char* GetRuleId() const = 0;
        virtual void Run(const DiagnosticContext& ctx,
                          std::vector<DiagnosticSuggestion>& out) const = 0;
    };

    struct DiagnosticContext
    {
        const ValidationContext     validation;   // de M100.48
        ActiveTool                  currentActiveTool;
        bool                        hasOpenedDialog;
        std::vector<std::string>    recentCommandHistory; // 10 dernières
        bool                        hasUserAttemptedExport;
        // ... etc.
    };
}
```

#### Partie D — Tests

`tutorial_diagnostic_tests` :
- Tutoriel : enchaînement des 8 étapes, validation correcte par étape, skip/resume.
- Diagnostic : injection d'un cas pour chaque règle MVP, vérification que la suggestion est émise.

### Hors scope

- **Plusieurs tutoriels** : MVP livre uniquement `first_launch.json`. Architecture supporte d'autres tutoriels, livrés plus tard.
- **Tutoriel adaptatif** : pas de personnalisation selon le profil utilisateur (dev vs artist).
- **Localisation > FR** : MVP en français uniquement.
- **Mode démo** : le tutoriel exécute les vraies actions, pas de mode sandbox.
- **Génération IA de suggestions** : toutes les règles sont codées en dur.
- **Replay de session** : pas d'enregistrement des actions utilisateur.
- **Tutoriel embarqué pour la phase 13** : sera couvert par un futur ticket.

## Schéma d'interaction utilisateur

### Tutoriel

```
[Premier lancement éditeur, pas de prefs.tutorialCompletionFlags]
  → Au 1er click utilisateur sur l'UI principale → modal d'accueil
  → "Bienvenue dans l'éditeur LCDLLN"

[Click "Lancer le tutoriel"]
  → TutorialSystem.Start("first_launch")

  → ZonePresetDialog.opened détecté via customValidator
  → TutorialSystem.AdvanceStep()
  → Étape 2 : surlignage Forêt tempérée

[Continue ainsi pour les 8 étapes]
  → Étape 3 : Créer la zone (ZonePresetExecutor lance exécution réelle)
  → Étape 4-6 : pose d'une cave (toolbar.cave → catalog → click viewport)
  → Étape 7 : validation (toolbar.validate)
  → Étape 8 : informationnel sur export

[Click "Quitter pour l'instant" à n'importe quelle étape]
  → tutorialCompletionFlags.firstLaunchProgress = currentStepIndex
  → Sequence avorted
  → Au prochain boot → modal propose "Reprendre"

[Fin tutoriel]
  → Modal de félicitations
  → Propose : "Essayer l'assistant pour personnaliser la prochaine zone ?"
  → Hook vers M100.50 Quick Start Wizard
```
