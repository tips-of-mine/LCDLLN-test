# M19.7 — Request/Response dispatcher + request_id tracking

**Status:** Closed

## Objectif
Implémenter un mécanisme request/response basé sur `request_id` (matching), avec support réponses out-of-order et server push.

## Dépendances
- M19.2
- M19.3
- M19.4
- M00.1

## Livrables
- Générateur request_id côté client
- Table pending requests:
  - timeout (5s)
  - callback/promise
- Routage:
  - response (request_id != 0) → pending
  - server push (request_id == 0) → handler push
- API:
  - `SendRequest(opcode, payload, onResponse, timeout)`
  - `SendPush(opcode, payload)` côté serveur

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- request_id monotonic ou pseudo-random; éviter collisions dans fenêtre pending.
- Timeout pending: 5s (configurable) → erreur TIMEOUT.

## Étapes d'implémentation
1. Générer request_id.
2. Stocker pending (map) + deadline.
3. À réception packet: matcher ou route push.
4. Dispatch callbacks sur thread main (pas thread réseau).
5. Tests out-of-order (3 requêtes simultanées).

## Critères de validation (DoD)
- [ ] Réponses out-of-order match correctement.
- [ ] Timeout 5s → pending nettoyé.
- [ ] Server push route sans pending.

## Notes / pièges à éviter
- Ne pas exécuter callback sur thread IO.
