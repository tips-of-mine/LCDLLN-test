# M100.44 — VMap Bridge & Phase 11 Validation (Clôture Phase 11)

## Résumé

**Ticket final qui boucle la dette technique cumulée de la Phase 2.5 et de la Phase 11** : le serveur n'a jamais reçu les édits de terrain de M100.35-39 (érosion ne propage pas vers `.vmap`), ni les mesh inserts de M100.40-43 (caves, overhangs, arches, donjons sont invisibles côté serveur pour collision/LOS). Conséquences gameplay actuelles :

- Un joueur peut tomber à travers une falaise dont la heightmap a été creusée par l'érosion (le serveur croit toujours le terrain plein).
- Un projectile peut traverser un mesh insert (le serveur ne voit pas la grotte / l'arche / la porte de donjon).
- Un PNJ peut « voir » à travers une montagne qui a été ouverte par un mesh insert.

Ce ticket livre :

1. **Le bridge éditeur → vmap** : lors de l'export d'une zone, un pipeline offline régénère les `.vmap` tiles correspondant aux chunks modifiés, en intégrant à la fois la heightmap mise à jour ET les meshes des mesh inserts comme géométrie de collision additionnelle.

2. **L'extension du format `.vmap`** pour porter les meshes additionnels : nouvelle section « AuxiliaryColliders » qui liste des références aux meshes glTF du content path, leur transform, et leur AABB.

3. **Le runtime serveur** (`src/shardd/internals/vmap/`) mis à jour pour charger ces colliders auxiliaires et les intégrer aux requêtes `IsInLineOfSight`, `GetHeight`, et `Raycast`.

4. **L'outil `lcdlln_vmap_extractor`** (existant via SERVER-CORE.05) mis à jour pour gérer les mesh inserts et la nouvelle section.

5. **La suite de tests d'intégration Phase 11** : 7 scénarios end-to-end qui simulent un joueur traversant chaque catégorie de volume 3D, et vérifient la cohérence collision/LOS client-server.

6. **Le bilan de la Phase 11** : doc `docs/PHASE_11_VOLUMES_3D.md` qui synthétise l'architecture, les invariants, le runbook de déploiement, et les limites connues.

**Un nouvel exécutable** : non — l'outil `lcdlln_vmap_extractor` existe déjà (SERVER-CORE.05), il est ici **étendu**, pas créé.

## Contexte critique à respecter

### 1. Le bridge est offline, pas runtime

Le bridge ne fonctionne **pas à chaud** lors d'éditions dans l'éditeur. Le workflow attendu est :

```
1. Éditeur : sauvegarder la zone (bundle export)
2. Outil offline : lcdlln_vmap_extractor --zone <bundle> --out vmap_tiles/
3. Déployer vmap_tiles/ vers le serveur (rsync, S3, etc.)
4. Redémarrer le shard responsable de la zone (ou hot-reload si supporté)
```

C'est volontaire : faire du hot-reload de collision sur un serveur live est extrêmement risqué et hors scope.

### 2. Couplage version `mesh_inserts.bin` ↔ `.vmap`

Si le client et le serveur n'ont pas la même version des données de collision, un joueur peut percevoir un mur que le serveur ne voit pas (ou vice versa). Le format `.vmap` étendu inclut un **hash de contenu** des `mesh_inserts.bin` source. Le runtime serveur vérifie ce hash au load et **refuse de servir** une zone si la version est incohérente avec la table `zone_content_versions` (qui sera mise à jour par le pipeline de déploiement).

## Portée

### Inclus

#### Partie A — Extension du format `.vmap`

Le format `.vmap` actuel (SERVER-CORE.05) contient une grille de heightmap server-side + zones de collision dérivées. Ce ticket ajoute :

##### A.1 — Nouvelle section "AuxiliaryColliders"

Append à la fin du `.vmap` tile :

```
[Existing sections...]

[AuxiliaryColliders section]
- Magic: "AUXC"
- Version: 1
- Count: uint32
- For each collider:
  - guid (16 bytes)
  - meshPathRelative (length-prefixed UTF-8, ex: "meshes/caves/cave_medium_01.gltf")
  - worldTransform (4×4 float32 row-major)
  - localAABB (Vec3 min + Vec3 max)
  - collisionProxyPath (length-prefixed UTF-8, ex: "meshes/caves/cave_medium_01.collision.bin")
  - flags: uint8 (bit 0 = blockLOS, bit 1 = blockMovement, bit 2 = isDungeonPortalSolid)
  - dungeonId (length-prefixed UTF-8, vide si pas un donjon)
```

##### A.2 — Hash de contenu

Header du `.vmap` étendu : nouveau champ `meshInsertsBinHash` (32 bytes, SHA-256) qui correspond au hash de l'`mesh_inserts.bin` source au moment du bake. Permet de détecter une incohérence si le client a une version + récente du `.bin` que le `.vmap` server-side.

##### A.3 — Bump version

`kVmapFormatVersion` passe de **N** à **N+1**. Migration : les anciens `.vmap` v(N) sont rejetés au load avec message clair (exigent re-bake).

#### Partie B — Extension de `lcdlln_vmap_extractor`

L'outil offline existant (SERVER-CORE.05) gagne :

##### B.1 — Nouveau flag `--zone-bundle`

```
lcdlln_vmap_extractor \
  --zone-bundle <path/to/zone_bundle> \
  --out <path/to/vmap_tiles> \
  [--threads 8]
```

Lit le bundle de zone (terrain.bin + mesh_inserts.bin + ...) et produit les `.vmap` tiles correspondants.

##### B.2 — Pipeline interne

1. Charger `terrain.bin` (M100.5).
2. Charger `mesh_inserts.bin` (M100.40).
3. Pour chaque tile vmap (typiquement 533 × 533 m, alignée chunk M100.5) :
   - Bake la heightmap server-side classique (SERVER-CORE.05).
   - Pour chaque `MeshInsertInstance` dont l'AABB chevauche la tile :
     - Charger le `MeshAsset` glTF.
     - Charger le collision proxy `.collision.bin` (M100.12).
     - Ajouter à la section AuxiliaryColliders.
4. Calculer le `meshInsertsBinHash` SHA-256 de l'`mesh_inserts.bin`.
5. Écrire le `.vmap` v(N+1) avec header + sections.

##### B.3 — Validation

L'outil émet des warnings si :
- Un mesh insert référence un glTF introuvable → warning + skip de cette instance.
- Un mesh insert a un collision proxy manquant → warning + utilise le mesh visual comme fallback (bbox de collision dégradée).
- Une tile reçoit > 50 mesh inserts → warning de perf (potentiel hotspot LOS query côté serveur).

#### Partie C — Runtime serveur

##### C.1 — Loader

`src/shardd/internals/vmap/VmapTileLoader.cpp` mis à jour :
- Vérifie `kVmapFormatVersion >= N+1`, sinon refuse de charger.
- Parse la section AuxiliaryColliders.
- Pour chaque collider : charge le `.collision.bin` du content path (cache shared).
- Enregistre les colliders dans `VmapTile::auxiliaryColliders`.

##### C.2 — Queries étendues

- `IsInLineOfSight(start, end, tile)` : raycast contre la heightmap AS BEFORE, **PLUS** raycast contre chaque collider auxiliaire dont l'AABB touche le segment. Renvoie `false` si **n'importe lequel** bloque.
- `GetHeight(x, z, tile)` : retourne la **hauteur la plus haute** entre la heightmap et la surface dessus du collider auxiliaire le plus proche (pour qu'un joueur sur un overhang ait la bonne hauteur sol).
- `Raycast(start, end, tile)` : raycast complet contre heightmap + colliders, retourne le hit le plus proche.

##### C.3 — Filtre par category

- Pour `IsInLineOfSight` : tous les colliders avec `flags.blockLOS = true` participent.
- Pour `GetHeight` / `Raycast` : tous les colliders avec `flags.blockMovement = true` participent.
- Cela permet à un mappeur de poser une « grotte avec parois transparentes au LOS » (ex : couloir de ventilation visuel) si jamais besoin spécifique.

##### C.4 — Hash mismatch handling

Au load d'une tile, si le `meshInsertsBinHash` ne correspond pas à celui que le master a annoncé via le check `zone_content_versions` : refuser la tile + log d'alerte. Le mappeur sait qu'il doit re-bake.

> **Déploiement** : ⚠️⚠️ **MAJEUR** — bump `kVmapFormatVersion` (N → N+1).

> **Étapes obligatoires** (lock-step) :
> 1. Lancer `lcdlln_vmap_extractor --zone-bundle <bundle> --out vmap_tiles/` pour chaque zone modifiée.
