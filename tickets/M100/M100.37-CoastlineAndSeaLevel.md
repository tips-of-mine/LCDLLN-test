# M100.37 — Coastline & Sea Level Editor

## Résumé

Outil éditeur qui définit **un niveau de mer global** pour la zone et matérialise tout ce qui en découle :

- **Une surface océan unique** générée automatiquement : un `LakePolygon` (format M100.13) qui couvre l'ensemble de la zone à hauteur `seaLevelMeters`. Le rendu eau M100.14 et la nage M100.15 s'appliquent gratuitement.
- **Visualisation interactive de la côte** : surlignage des cellules dont la heightmap traverse le niveau de mer, recalculée en temps réel quand l'utilisateur ajuste le slider.
- **Génération optionnelle de bandes côtières** : peinture automatique d'une couche splat « sable » (M100.10 format) sur les cellules en bordure de mer, jusqu'à une largeur paramétrable.
- **Smoothing optionnel de la côte** : lissage Gaussien limité aux cellules dans une bande autour de l'altitude `seaLevelMeters ± thresholdMeters`, pour atténuer les transitions abruptes terre-mer (effet « plage adoucie »).
- **Génération optionnelle de falaises côtières** : un pass qui détecte les cellules à pente locale > seuil **et** proches du niveau de mer, et augmente leur verticalité (élève les cellules juste au-dessus du niveau, abaisse celles juste en-dessous).

L'outil partage le **niveau de mer** avec M100.36 : `seaLevelMeters` est déjà centralisé dans `WaterDocument::GetOcean().seaLevelMeters` depuis M100.36 (qui a introduit la struct minimale `OceanSettings`). Ce ticket **enrichit** la struct avec les nouveaux champs visuels et atmosphériques (`bottomColor`, `turbidity`, `windInfluence`, `enabled`) — pas de refactor, ajout de champs uniquement. Modifier la valeur du sea level via cet outil met à jour la simulation hydrologique au prochain Simulate.

L'outil `CoastlineEditorTool` est enregistré dans la toolbar M100.35. Tout est packagé en une `ICommand` réversible par invocation **Apply**.

**Aucun nouvel exécutable n'est créé.** Code dans `engine_core`, binaire éditeur unique `world_editor_app` (`lcdlln_world_editor.exe`).

## Contexte critique à respecter

### 1. `OceanSettings` est déjà centralisé — ce ticket enrichit, ne refactore pas

`OceanSettings { seaLevelMeters }` a été introduit par M100.36 et est, depuis ce ticket, la source de vérité unique du sea level de la zone (stockée dans `WaterDocument::m_ocean`, persistée dans `instances/water.bin` v2).

Ce ticket **étend** la struct avec quatre nouveaux champs (`bottomColor`, `turbidity`, `windInfluence`, `enabled`) et bumpe `kWaterBinVersion` v2 → v3. Aucun champ n'est déplacé, aucun appelant n'est cassé :

- `WatershedSimulationParams` reste sans `seaLevelMeters` (déjà retiré en M100.36).
- `RunWatershedSimulation(TerrainDocument&, const WaterDocument&, const WatershedSimulationParams&)` continue de lire le sea level via `water.GetOcean().seaLevelMeters`. Aucun changement de signature.
- L'UI de M100.36 continue d'afficher son slider sea level qui édite la même `OceanSettings`. L'UI de ce ticket peut éditer le sea level **et** les nouveaux champs.
- La `CoastlineCommand` snapshotte les `OceanSettings` complètes (anciennes / nouvelles) pour Undo/Redo.

### 2. Cohérence avec M100.13

- L'océan **est** un `LakePolygon` standard, juste avec un polygone englobant la zone entière + une marge externe de 1 000 m pour éviter les artefacts de bord de rendu.
- `LakePolygon::isOcean = true` ajouté au struct M100.13 pour permettre au rendu M100.14 d'appliquer (futur) un comportement différent : pour ce ticket, comportement identique aux lacs, le flag est juste informatif.
- Un seul `LakePolygon` d'océan par zone. Si l'utilisateur applique deux fois la `CoastlineCommand`, la seconde remplace la première dans `WaterDocument`.

### 3. Invariant terrain visible (rappel M100.35)

- Pas de bascule caméra automatique.
- Le plan visuel sea level (overlay bleu plan horizontal au niveau Y) est un mesh additif blended, **lecture seule** sur depth.
- Le surlignage des cellules côtières est un overlay 2D draw-list, pas un render pass.
- L'outil reste fonctionnel quand le terrain est en fallback orange.

### 4. Une seule application

Tout vit dans `engine_core`. Pas de `add_executable` autre que la suite de tests `coastline_tests`.

### 5. Arbo server-core-style

- `src/world_editor/water/`, `src/world_editor/terrain/`, `src/world_editor/splat/`, `src/world_editor/ui/`, `src/world_editor/tests/`.
- Includes absolus depuis la racine.

#### Partie C — Génération océan

À l'**Apply**, si aucun océan n'existe encore dans `WaterDocument` :
1. Crée un `LakePolygon` avec :
   - polygone = rectangle englobant la zone + marge externe 1 000 m (5 vertices, sens horaire),
   - niveau d'eau = `seaLevelMeters`,
   - `isOcean = true`,
   - `bottomColor`, `turbidity`, `flowSpeed` = valeurs UI.
2. Insère dans `WaterDocument`.

Si un océan existe déjà :
- Met à jour ses paramètres in-place (niveau, couleur, turbidité) sans recréer le polygone.

#### Partie D — Smoothing côtier (optionnel)

Si **Lisser la côte** est coché à l'Apply :
- Pour chaque cellule dans la bande `[seaLevelMeters - bandeMeters, seaLevelMeters + bandeMeters]` :
  - Applique un kernel Gaussien 3×3 lecture-seule (lit la heightmap consolidée pré-edit), force `force`.
  - Écrit le delta `(h_lissé - h_original)` dans `SparseChunkDeltas`.
- Multi-chunks, parité de couture identique à M100.35.

#### Partie E — Plage automatique (optionnel)

Si **Peindre une bande de sable** est coché à l'Apply :
- Pour chaque cellule entre `[seaLevelMeters - largeurMer, seaLevelMeters + largeurTerre]` (mesuré verticalement, pas en distance horizontale) :
  - Calcule un poids `w = smoothstep(largeur_meters, 0, abs(altitude_cellule - seaLevelMeters))`.
  - Atténué par `attenuationFactor` au bord (la cellule la plus éloignée de la côte reçoit `w *= (1 - attenuationFactor)`).
  - Écrit le poids dans le splat de la couche sélectionnée (`SplatLayer::Sand` par exemple, sélectionnée dans l'UI).
- L'écriture splat est cumulative (additive) pour ne pas effacer les peintures existantes — la couche sable s'ajoute à ce qui est déjà là.

#### Partie F — Falaises côtières (optionnel)

Si **Générer des falaises** est coché :
- Pour chaque cellule dans la bande `[seaLevelMeters - thresholdMeters, seaLevelMeters + thresholdMeters]` (utilise la même bande que le smoothing) :
  - Calcule la pente locale `slope` = `|grad(heights)| * cellSize` au point.
  - Si `slope > slopeThresholdDeg` :
    - Si altitude > sea level : `delta = +cliffLandSideMeters * smoothstep(...)`.
    - Si altitude < sea level : `delta = -cliffSeaSideMeters * smoothstep(...)`.
  - Sinon : pas de modification.
- Cumulé avec le smoothing (les deux deltas s'additionnent en `SparseChunkDeltas`).

#### Partie G — Commit unique

À l'**Apply**, une seule `CoastlineCommand` est poussée. Elle contient :
- les paramètres océan à appliquer dans `WaterDocument::OceanSettings`,
- l'éventuel `LakePolygon` à insérer (si océan n'existait pas),
- les deltas heightmap (smoothing + falaises),
- les deltas splat (plage auto).

Undo restaure tout octet-à-octet.

#### Partie H — Tests

Suite `coastline_tests` couvrant : génération océan, surlignage marching squares, smoothing limité à la bande, plage auto sur sable, falaises, undo complet, lien avec M100.36.

### Hors scope

- **Niveau de mer variable dans le temps** (marées, tempêtes) → M100.26 weather system.
- **Multiples niveaux de mer** dans une même zone (deux océans à altitudes différentes). Un seul océan global.
- **Falaises avec surplomb** (overhangs) — impossible avec heightmap 2D, Phase 11 (mesh inserts).
- **Grottes marines / sea caves** — Phase 11.
- **Récifs coralliens, herbiers** — futur ticket vegetation aquatique.
- **Profondeur variable de l'océan via heightmap** — l'océan est plat à `seaLevelMeters`. Si la heightmap sous la mer descend très bas, le rendu eau M100.14 affichera la couleur de fond + turbidité, mais aucun underwater detail mesh.
- **Bandes côtières non-splat** : la « plage » est juste une couche splat sable. Pas de génération de mesh galets, pas de prop sable (coquillages, etc.).
- **Estuaires / zones de mélange eau douce-salée** : les rivières M100.36 qui arrivent à la mer terminent en spline simple ; pas de zone tampon.
- **Bridge vmap** : la nouvelle surface océan ne propage pas aux tiles `.vmap` server-side. Inchangé par rapport à M100.13.
- **Persistance des paramètres outil non océanographiques** (force smoothing, largeurs plage, etc.) : non sauvegardés, valeurs par défaut au boot.

## Spécification fonctionnelle

### Workflow

1. Cliquer **Coastline** dans la toolbar.
2. Deux overlays apparaissent : le plan sea level bleu transparent + le trait jaune des côtes courantes.
3. Ajuster le slider **Sea level (Y)** : les deux overlays bougent en temps réel. Les statistiques (surface terre / surface océan / longueur côte) se mettent à jour avec debounce 200 ms.
4. Configurer optionnellement : smoothing, plage auto, falaises.
5. Cliquer **Apply** → `CoastlineCommand` poussée.
6. Le rendu M100.14 affiche maintenant l'océan ; les côtes ont reçu leur splat de sable (si activé) ; la heightmap a été lissée / falaisée (si activé).

### Algorithme — marching squares pour la côte

Pour chaque carré de 2×2 cellules adjacentes du grid heightmap :
- Classifier chaque coin : `1` si `altitude > seaLevelMeters`, `0` sinon.
- Code 4 bits → 16 cas (dont 2 triviaux : tout terre / tout mer = pas de segment).
- Pour les 14 cas non triviaux : tracer 1 ou 2 segments interpolés linéairement entre les arêtes du carré.
- Empile les segments en world space → rendu en `ImGui::GetForegroundDrawList()`.

Optimisation : la liste de segments est cached et recalculée seulement quand `seaLevelMeters` change ou la heightmap a été modifiée depuis la dernière passe.

### Algorithme — statistiques en live

- Surface terre : `Σ (altitude > seaLevelMeters) / N_total`.
- Surface océan : complément.
- Longueur côte : somme des longueurs des segments marching squares.
- Cellules de plage : compte des cellules dans la bande verticale `[sea - largeurMer, sea + largeurTerre]`.

Toutes les statistiques sont O(N) sur le grid consolidé. À 5121² cellules, ~30 ms single-thread, dans le budget 200 ms de debounce.

### Génération du polygone océan

```
oceanPolygon = [
    (-1000,         -1000),
    (kZoneSize+1000,-1000),
    (kZoneSize+1000, kZoneSize+1000),
    (-1000,          kZoneSize+1000),
    (-1000,         -1000),  // ferme
]
```

5 vertices, sens horaire, en mètres dans l'espace zone. Triangulé par `LakePolygon` (M100.13) en 4 triangles. Coût rendu négligeable.

### Couture inter-chunks

Smoothing, plage, falaises produisent tous des `SparseChunkDeltas` / `SparseSplatDeltas` en espace monde global → couture inter-chunks préservée par construction. Tests dédiés.

### Performance attendue

- Marching squares sur 5121² : ~25 ms single-thread.
- Statistiques live : ~30 ms.
- Smoothing 3×3 dans la bande (typiquement 5-10 % des cellules) : ~80 ms.
- Plage auto + falaises : ~50 ms.
- Total Apply : **< 500 ms** pour une zone complète.

### Format / parité

- `instances/water.bin` : **bump `kWaterBinVersion` v2 → v3**. Deux changements :
  1. Le champ `LakePolygon::isOcean` (1 octet par instance lac).
  2. La section « ocean » est étendue de 4 octets (v2 : `seaLevelMeters` seul) à ~32 octets (v3 : sea level + couleur + turbidité + windInfluence + enabled).
  Le reader v3 lit aussi v2 et v1 ; les nouveaux champs reçoivent leurs valeurs par défaut.
- `terrain.bin` : modifié par smoothing + falaises (M100.5 inchangé).
- `splat.bin` : modifié par plage auto (M100.9 inchangé).
- **Pas de changement de format JSON** `map.lcdlln_edit.json`.

## Schéma d'interaction utilisateur

```
[click bouton « Coastline » dans la toolbar]
  → m_activeTool = Coastline
  → overlay plan sea level bleu (alpha 0.25) apparaît à Y=50
  → overlay côte jaune (marching squares) trace les contours

[drag slider Sea level]
  → plan bleu et trait jaune bougent en temps réel
  → debounce 200 ms : statistiques recalculées
    (surface terre/mer, longueur côte)

[tick checkbox « Peindre une bande de sable »]
  → preview overlay supplémentaire : surlignage cellules dans la bande
    [sea - largeurMer, sea + largeurTerre]
  → compteur "cellules de plage" mis à jour

[tick checkbox « Lisser la côte » / « Falaises »]
  → preview supplémentaire : surlignage des cellules qui seront modifiées
  → diffusion en debounce des deltas potentiels (juste un compteur,
    pas de simulation full)

[click Apply]
  → CoastlineCommand pushée :
      WaterDocument.SetOceanSettings(...) + insertion LakePolygon océan
      TerrainDocument.ApplyDeltas(smoothing + falaises)
      SplatDocument.ApplyDeltas(plage)
      EnqueueLodRegen pour chunks touchés
  → overlays disparaissent, l'outil reste actif

[Ctrl+Z]
  → Undo : restore OceanSettings précédentes,
    retire le LakePolygon (s'il a été créé),
    inverse tous les deltas heightmap + splat
```

## Critères d'acceptation

### Overlays interactifs

- [ ] Activer l'outil affiche le plan sea level (mesh blended bleu alpha 0.25) à `Y = seaLevelMeters`.
- [ ] Le trait côtier jaune (marching squares) trace correctement l'iso-altitude `seaLevelMeters` à ±0.5 m (tolérance d'interpolation linéaire entre cellules).
- [ ] Bouger le slider sea level met à jour les deux overlays dans la même frame (sans debounce visible).
- [ ] Les statistiques affichées (surface terre, surface océan, longueur côte) sont rafraîchies avec debounce 200 ms, pas plus.

### Génération océan

- [ ] Apply sur une zone sans océan crée **un** `LakePolygon` avec `isOcean = true`, polygone englobant la zone + marge 1 000 m, niveau d'eau = `seaLevelMeters`.
- [ ] Apply sur une zone avec océan existant met à jour les paramètres in-place, ne crée pas de second `LakePolygon`.
- [ ] L'océan créé est rendu par M100.14 (Water Render Pass) avec couleur de fond et turbidité corrects.

### Smoothing / falaises / plage

- [ ] Avec **Lisser la côte** coché, les cellules dans `[sea-band, sea+band]` reçoivent un delta lecture-stable (lit la heightmap pré-edit, écrit après).

- [ ] Avec **Peindre une bande de sable** coché, le splat de la couche sélectionnée reçoit des poids `> 0` uniquement dans la bande verticale ; les autres couches splat restent intactes.
- [ ] Tous les deltas sont additifs en cas de Apply répétés (sauf l'océan qui est in-place).
