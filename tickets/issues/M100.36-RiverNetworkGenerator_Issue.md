## Contenu du ticket (M100.36)

# M100.36 — River Network Generator (Watershed Simulation)

## Résumé

Génère automatiquement un réseau de rivières cohérent à partir de la heightmap actuelle et de **sources** (springs) posées par l'utilisateur sur le terrain. La simulation utilise l'algorithme **D8 flow direction** (chaque cellule s'écoule vers le voisin de plus grande pente descendante) combiné à un **flow accumulation** pour identifier les chemins d'écoulement. Les chemins issus des sources sont tracés en descendant jusqu'à :

- le **niveau de mer global de la zone** (`WaterDocument::GetOcean().seaLevelMeters`) → terminaison par embouchure,
- un **minimum local** (sink ; terminaison par lac auto-généré optionnel),
- ou la **frontière de la zone** (terminaison silencieuse, l'écoulement hors-zone n'est pas modélisé).

Ce ticket **introduit la struct `OceanSettings`** (champ minimal `seaLevelMeters`) comme source de vérité unique de la zone, stockée dans `WaterDocument::OceanSettings` et persistée dans `instances/water.bin` (bump `kWaterBinVersion` v1 → v2). M100.37 enrichira cette struct (couleur de fond, turbidité, vagues) **sans refactoring** : ajout de champs uniquement, pas de déplacement.

Les chemins cellule-à-cellule sont simplifiés en **splines** via Douglas-Peucker puis convertis en **instances RiverTool de M100.13** : le rendu, la collision et le gameplay de l'eau réutilisent intégralement la pile existante. Optionnellement, l'application **carve** la heightmap le long du tracé pour creuser un lit de rivière visible.

L'outil `RiverNetworkTool` est enregistré dans la toolbar à icônes livrée par M100.35. Tout est packagé en une seule `ICommand` réversible.

**Aucun nouvel exécutable n'est créé.** Le code vit dans `engine_core`, linké au binaire éditeur unique `world_editor_app` (output `lcdlln_world_editor.exe`).

## Contexte critique à respecter

### 1. Cohabitation avec M100.13

`RiverTool` (M100.13) accepte une `Spline3D` + paramètres d'eau (turbidité, couleur de fond, vitesse de flot). `RiverNetworkTool` **produit ces mêmes structures** et les insère dans `WaterDocument` exactement comme si l'utilisateur avait tracé chaque rivière à la main avec `RiverTool`. Aucun nouveau format binaire, aucun nouveau code de rendu. Le réseau est donc déjà supporté par M100.14 (Water Render Pass) et M100.15 (Water Surface Hook nage).

### 3. Une seule application

## Portée

### Inclus

#### Partie A — Pose de sources

- Nouveau bouton-icône **River Network** dans la toolbar M100.35 (`game/data/icons/editor/tools/river_network.png`).
- Curseur croix-de-mire au-dessus du viewport quand l'outil est actif.
- **Clic gauche** sur le terrain : pose une source (`SpringSource`) à la position raycastée. Marqueur visuel : goutte bleue 8 px + halo.
- **Clic droit** sur une source existante (rayon de 12 px) : la retire.
- **Drag** sur une source : la déplace.
- Liste des sources affichée dans `ToolPropertiesPanel` (coordonnées X / Z + bouton supprimer par ligne).
- Limite : 64 sources par session (au-delà : tooltip d'erreur, source non ajoutée).

#### Partie B — Simulation D8 + flow accumulation

- Bouton **Simulate** dans `ToolPropertiesPanel`. Bloque l'UI avec une modale de progression (« Simulation hydrologique en cours… ») pendant l'exécution.
- Algorithme :
  1. **Consolidation heightmap** : assemble un tableau lecture-seule des hauteurs de tous les chunks chargés (résolution `5121×5121` pour une zone complète, soit ~104 MB en `float`). Si la zone n'est pas entièrement résidente, charger les chunks manquants via `TerrainDocument::EnsureChunkResident(coord)`.
  2. **D8 flow direction** : pour chaque cellule, calcule le voisin (parmi les 8) de plus grande pente descendante. Tie-break déterministe : ordre NE → E → SE → S → SW → W → NW → N. Cellule sans voisin descendant = sink (encoding spécial `kSinkDir = 255`).
  3. **Flow accumulation** : tri topologique des cellules par altitude décroissante, puis propagation cumulative. Chaque cellule reçoit `acc = 1 + Σ (acc des cellules amont qui s'écoulent vers elle)`.
  4. **Tracé des chemins depuis sources** : pour chaque `SpringSource`, suivre la direction D8 jusqu'à atteindre soit :
     - sea level (altitude ≤ `WaterDocument::GetOcean().seaLevelMeters`) → terminaison embouchure,
     - sink (`dir == kSinkDir`) → terminaison + lac auto si `autoLakesAtSinks = true`,
     - frontière zone → terminaison silencieuse,
     - cellule déjà visitée par un autre chemin → confluence (le chemin amont fusionne avec le chemin aval).
  5. **Filtrage par flow threshold** : un segment dont le `flowAcc` max est < `minFlowThreshold` est rejeté (rivière trop maigre).
  6. **Simplification Douglas-Peucker** : chaque chemin cellule-à-cellule est réduit à une `Spline3D` (≤ ~50 points par rivière, tolérance `simplificationToleranceMeters`).

#### Partie C — Preview interactive

- Preview overlay : lignes bleues semi-transparentes sur le viewport, épaisseur proportionnelle à `log(flowAcc)`, opacité 0.7.
- Marqueurs aux confluences (cercle bleu 6 px).
- Marqueurs aux embouchures (triangle bleu).
- Marqueurs aux sinks avec `autoLakesAtSinks = true` (cercle bleu rempli).
- Compteurs dans `ToolPropertiesPanel` : « N rivières détectées · M confluences · K embouchures · L lacs auto ».
- Recalcule la preview au tweak de **n'importe quel paramètre** (debounce 200 ms), sauf modification des sources qui invalide le résultat → bouton **Simulate** redevient nécessaire.

#### Partie D — Carving optionnel de la heightmap

- Checkbox **Carve heightmap along rivers** dans `ToolPropertiesPanel`. Si coché à l'**Apply** :
  - Pour chaque chemin de rivière simplifié, applique un profil gaussien de creusement : `depth(d) = carveDepthMeters * exp(-2 * (d / carveWidthMeters)^2)` où `d` est la distance perpendiculaire à la spline.
  - Multi-chunks : même pattern que M100.35, deltas sparse + couture inter-chunks préservée.
- Si décoché : seules les rivières (splines + données eau) sont créées. La heightmap reste intacte. Risque visuel : rivière qui « flotte » au-dessus du terrain si pente locale > pente moyenne. Tooltip d'avertissement affiché.

#### Partie E — Lacs auto aux sinks

- Si **Auto-lakes at sinks** est coché ET un chemin se termine sur un sink :
  - Calcule le bassin d'écoulement local : flood-fill 8-connexité depuis le sink jusqu'à atteindre l'altitude `sinkAltitude + autoLakeMaxDepthMeters` ou l'altitude où l'eau déborderait par une cellule de la frontière (cellule la plus basse de la frontière du bassin).
  - Crée une `LakePolygon` (M100.13 format) avec :
    - polygone = frontière du bassin (simplifiée Douglas-Peucker),
    - niveau d'eau = `min(sinkAltitude + autoLakeMaxDepthMeters, débordement)`,
    - couleur de fond et turbidité = valeurs par défaut LakeTool.

#### Partie F — Commit unique

- L'**Apply** crée une seule `RiverNetworkCommand` qui contient :
  - les `Spline3D` à ajouter dans `WaterDocument`,
  - les `LakePolygon` (si `autoLakesAtSinks`),
  - les `SparseChunkDeltas` de carving (si `carveHeightmap`).
- Undo : retire toutes les rivières et lacs ajoutés ET inverse les deltas de carving.

#### Partie G — Tests

- Suite `river_network_tests` couvrant : D8 sur damiers, flow accumulation déterministe, tracé jusqu'à mer / sink / bord, confluence de 2 sources, Douglas-Peucker, carving multi-chunks, undo complet.

### Hors scope

- **Simulation hydraulique réelle** : pas de calcul de débit en m³/s, pas de gestion d'évaporation, pas de variation saisonnière. D8 est une approximation suffisante pour la génération géométrique.
- **Distinction main river vs tributary** : la `RiverNetworkCommand` produit des `Spline3D` plates, sans hiérarchie de tributaires nommée. Pour gérer des « grands fleuves nommés » vs des « affluents », l'utilisateur applique manuellement des tags via Inspector après commit.
- **Deltas d'embouchure** : la rivière se termine en spline simple au niveau de la mer. Pas de géométrie de delta avec bras multiples.
- **Rivières souterraines** : aucune intersection avec les futures grottes M100.40+ ; la simulation se fait uniquement sur la heightmap 2D.
- **Variations météo / saisons** : M100.26 et M100.25 gèrent les modifs de surface eau dans le temps (gel hivernal, crue printanière). Non couvert ici.
- **Estuaires et marées** : pas de zone de mélange eau douce / salée, pas de niveau de mer variable.
- **Sources souterraines / résurgences** : toutes les sources sont en surface, posées sur la heightmap.
- **Streaming / chargement partiel** : si la zone n'est pas chargée intégralement en RAM, l'outil charge les chunks manquants pendant la simulation (bloquant). Pas de mode incrémental.
- **Multi-zone (rivières inter-zones)** : un chemin qui sort de la zone est tronqué à la frontière. Le raccordement avec la zone voisine est manuel.
- **Bridge vmap server-side** : les rivières créées ne sont pas répliquées dans les tiles `.vmap`. Le serveur ignore les surfaces d'eau (`SurfaceQueryService` côté client uniquement, M100.15). Pas de changement ici.
- **Édition post-Apply des rivières** : une fois la `RiverNetworkCommand` appliquée, chaque rivière devient une instance `RiverTool` standard, éditable individuellement via M100.13 mais plus comme un réseau cohérent. Re-générer le réseau = clear puis re-simulate.
- **Persistance des `SpringSource` et paramètres de simulation** : non sauvegardés. Re-ouvrir la map vide la liste des sources. Future extension possible : `world_editor/maps/<zone_id>/river_simulation.json`.

## Spécification fonctionnelle

### Workflow

1. Cliquer le bouton **River Network** dans la toolbar M100.35.
2. Cliquer sur le terrain : pose une source. Répéter pour N sources (typiquement 3-10 dans les zones de montagne).
3. Configurer les paramètres dans `ToolPropertiesPanel` (sea level, flow threshold, carving, lacs auto, etc.).
4. Cliquer **Simulate** → modale de progression (1-5 s pour une zone complète). À la fin, la preview bleue apparaît sur le viewport.
5. Tweak paramètres → preview recalculée (debounce 200 ms). Ajout / déplacement de source → re-cliquer **Simulate**.
6. Cliquer **Apply** → `RiverNetworkCommand` poussée. Les rivières apparaissent dans `WaterDocument`, le rendu M100.14 les affiche. Le panneau Outliner liste maintenant N entrées « River #i » sous Water surfaces.
7. Cliquer **Cancel** ou changer d'outil → abandon, sources et preview supprimées (avec confirmation modale si sources > 0).

### Algorithme D8 — pseudo-code

```
Pour chaque cellule (x, z) du grid heightmap consolidé :
  h_self = heights[x, z]
  best_dir = kSinkDir
  best_slope = 0
  Pour chaque (dx, dz) dans les 8 voisins (ordre NE → E → SE → S → SW → W → NW → N) :
    h_n = heights[x+dx, z+dz]
    delta = h_self - h_n
    dist = sqrt(dx*dx + dz*dz) * cellSizeMeters
    slope = delta / dist
    Si slope > best_slope :
      best_slope = slope
      best_dir = encode(dx, dz)
  flowDirection[x, z] = best_dir
```

### Algorithme flow accumulation

```
Trier toutes les cellules par altitude décroissante : sortedIdx[N]
Initialiser flowAcc[N] à 1.

Pour chaque idx dans sortedIdx :
  (x, z) = decode(idx)
  dir = flowDirection[x, z]
  Si dir != kSinkDir :
    (dx, dz) = decodeDir(dir)
    idx_down = encode(x + dx, z + dz)
    flowAcc[idx_down] += flowAcc[idx]
```

### Algorithme tracé depuis sources

```
visited = ensemble vide
paths = []
Pour chaque source dans springSources :
  (x, z) = cellule de la source
  Tant que vrai :
    Ajouter (x, z) au path courant
    Si visited contient (x, z) : confluence avec un path précédent. Break.
    Ajouter (x, z) à visited
    dir = flowDirection[x, z]
    Si dir == kSinkDir : sink. Break.
    Si heights[x, z] <= ocean.seaLevelMeters : embouchure. Break.   // ocean = waterDoc.GetOcean()
    (dx, dz) = decodeDir(dir)
    nx, nz = x + dx, z + dz
    Si hors zone : frontière. Break.
    x, z = nx, nz
  Ajouter path à paths si len(path) >= 2.
```

### Filtrage et simplification

- Pour chaque path : si `max(flowAcc[cells in path]) < minFlowThreshold`, rejeter.
- Sinon, simplifier par Douglas-Peucker à tolérance `simplificationToleranceMeters` → liste de points 3D (XZ depuis cellules, Y depuis heightmap).
- Construire une `Spline3D` (M100.13 format) à partir de cette liste.

### Couture inter-chunks (carving)

Identique à M100.35 : `SparseChunkDeltas` indexé par `GlobalChunkCoord`, écriture en espace monde global → les colonnes de bord partagées entre chunks adjacents reçoivent des deltas identiques. Test de parité au commit.

### Performance attendue

- D8 sur 5121² ≈ 26 M cellules : ~150 ms single-thread (boucle simple, 9 fmadd/cellule).
- Tri topologique : ~100 ms (radix sort sur altitude quantifiée 16-bit).
- Flow accumulation : ~80 ms (parcours unique post-tri).
- Tracé chemins (typiquement 4-10 sources × < 5000 cellules) : ~5 ms.
- Douglas-Peucker : négligeable.
- Carving sparse deltas : 50-200 ms selon nombre de chunks touchés.

Total cible : **< 1 s sur une zone complète** sur un Ryzen 5 5600 ou équivalent. Si > 5 s observés, paralléliser la phase D8 avec `engine::core::jobs::JobSystem` (cellules indépendantes).

### Format / parité

- **Bump `kWaterBinVersion` v1 → v2** : ajout d'une section « ocean » de taille fixe (4 octets pour `seaLevelMeters` en float32). Le reader v1 est conservé pour compatibilité descendante : `m_ocean` est initialisé à `OceanSettings{}` (sea level 50 m par défaut). Le writer écrit toujours en v2.
- Les rivières et lacs auto sont écrits dans `instances/water.bin` (M100.13 format v2) via `WaterDocument`.
- Le carving modifie `terrain.bin` (M100.5).
- **Pas de changement de format JSON** `map.lcdlln_edit.json`.
- Les `SpringSource` et `WatershedSimulationParams` sont **non persistés** (perdus à la fermeture de la map). Le `seaLevelMeters` édité dans le slider l'est, parce qu'il vit dans `OceanSettings` persisté.

## Critères d'acceptation

### Pose de sources

- [ ] Cliquer dans le viewport pose une `SpringSource` à la position raycastée (vérifié XZ ±0.5 m, Y égal à `TerrainChunk::SampleHeight(x, z)`).
- [ ] Cliquer-droit sur une source dans un rayon de 12 px la retire.
- [ ] Drag d'une source la déplace en temps réel (la preview est invalidée → bouton **Simulate** redevient le seul moyen de la recalculer).
- [ ] Au-delà de 64 sources, le 65ᵉ clic est ignoré et affiche un tooltip d'erreur.

### Simulation

- [ ] **Simulate** sur une zone vide (heightmap plate à Y=100) ne produit aucune rivière (`rivers.empty()`).
- [ ] **Simulate** avec sources placées dans des montagnes et `WaterDocument::GetOcean().seaLevelMeters = 50` produit au moins une rivière par source dont la flow accumulation max est ≥ `minFlowThresholdCells`.
- [ ] Deux sources qui convergent géométriquement vers la même vallée produisent **une seule rivière en aval** (confluence détectée).
- [ ] Une source posée dans un sink (cellule sans voisin descendant) produit **0 rivière** mais **1 lac auto** si `autoLakesAtSinks = true`.
- [ ] Le tracé d'une rivière depuis une montagne jusqu'au niveau de mer termine par une embouchure (cellule finale avec altitude ≤ `seaLevelMeters`).
- [ ] La simulation est **déterministe** : mêmes sources, mêmes paramètres, même heightmap → mêmes `RiverNetworkResult` byte-equal.

### Preview

- [ ] La preview overlay est rendue en lignes bleues semi-transparentes sur le viewport.
- [ ] Les confluences sont marquées par cercles, les embouchures par triangles, les lacs auto par cercles remplis.
