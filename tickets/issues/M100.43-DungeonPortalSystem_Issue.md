## Contenu du ticket (M100.43)

# M100.43 — Dungeon Portal System (Phase 11 — Volumes 3D)

## Résumé

Outil spécialisé pour poser des **entrées de donjons** : meshes glTF d'entrée (porte massive, ouverture cintrée, passage taillé) **accompagnés d'un trigger gameplay** qui, à l'entrée du joueur dans un volume sphérique au seuil, lance une **transition serveur vers une instance dédiée** (donjon = monde séparé, instancié).

Cette spécialisation diffère fondamentalement des grottes / overhangs / arches :

- **Pas de volume intérieur jouable** dans le monde extérieur : l'intérieur du donjon n'existe pas dans la zone overworld. Le mesh insert est seulement la « façade » de l'entrée.
- **Trigger réseau** : le seuil pose un `DungeonPortalTrigger` qui appelle `MasterShardClientFlow::RequestInstanceEnter(dungeonId)` quand le joueur le franchit (validation + spawn dans l'instance dédiée).
- **Métadonnées gameplay riches** : `dungeonId`, niveau recommandé, taille de groupe, faction restriction, jour/nuit lock, statut de complétion par compte.
- **Visibilité conditionnelle** : un portail de donjon peut être **caché** au joueur tant qu'une quête n'est pas accomplie.

Catégorie `insertCategory = "dungeon"`.

C'est le premier ticket de la phase 11 qui **touche le serveur** : nouveau opcode `INSTANCE_ENTER_REQUEST/RESPONSE`, table `dungeons` en DB.

**Un seul nouveau binaire évité** : tout reste dans `engine_core` côté client, dans `src/shardd/` côté shard.

## Dépendances

- **M100.40** — Mesh Insert Foundation (réutilisation totale).

## Contexte critique à respecter

### 1. Premier ticket réseau de la Phase 11

Tous les tickets M100.40-42 étaient client/éditeur-only. **M100.43 introduit du serveur** :

- Nouveau **opcode `INSTANCE_ENTER_REQUEST`** (client → master).
- Nouveau **opcode `INSTANCE_ENTER_RESPONSE`** (master → client : succès + endpoint shard d'instance, ou rejet).
- Nouvelle table DB `dungeons` (id, name_loc, recommended_level_min/max, party_size_min/max, etc.).
- Nouvelle table DB `dungeon_instance_state_per_account` (pour le statut de complétion par joueur, si applicable).

**Bump `kProtocolVersion`** : opcode allocation + déploiement serveur obligatoire en lock-step avec le client.

### 2. Pas d'implémentation gameplay du donjon lui-même

Ce ticket livre **uniquement l'entrée** : le mesh insert + le trigger + la requête réseau + la réponse master. La logique de spawn dans l'instance (création du shard, allocation de l'instance, peuplement des mobs) sera couverte par un futur ticket d'**InstanceManager runtime** (SERVER-CORE.19 + tickets dédiés). Pour ce ticket, l'`INSTANCE_ENTER_RESPONSE` peut renvoyer `"not_implemented"` et le client affiche une erreur UI propre — c'est volontaire.

### 3. Lien avec le bridge vmap

L'entrée du donjon est rendue par le mesh insert, et **le serveur doit savoir** qu'elle est solide pour les collisions (sinon le joueur passe à travers la porte). M100.44 livrera ce bridge. En attendant : warning UI dans `ToolPropertiesPanel`.

### 4-7. Invariant terrain visible, app unique, arbo, vmap

Identiques.

#### Partie B — Côté client (runtime jeu)

##### B.1 — Format `instances/dungeons.bin`

Indexé par chunk, contient les métadonnées des portails + la AABB de threshold. Pas les meshes (déjà dans `mesh_inserts.bin`).

##### B.2 — Trigger runtime

À chaque tick :
- Calcule la position du joueur.
- Pour chaque `DungeonPortal` dans le ring `Active` du chunk courant :
  - Si le joueur est dans le `thresholdAABB` ET pas déjà en transition :
    - Affiche une **bulle interactive** : « Entrer dans : Ruine antique de Selemar [F] ».
    - Si le joueur appuie sur la touche d'interaction : envoie `INSTANCE_ENTER_REQUEST` au master.

##### B.3 — Réception `INSTANCE_ENTER_RESPONSE`

- Si succès : sauve la position actuelle via `CHARACTER_SAVE_POSITION` (Phase 3.6.6), puis re-init `GameplayUdpClient` vers l'endpoint shard d'instance.
- Si rejet (faction, niveau, groupe, quête manquante, instance pleine, etc.) : affiche un toast UI avec la raison.

```cpp
struct InstanceEnterResponsePayload
{
    enum class Status : uint8_t
    {
        Success        = 0,
        NotFound       = 1,
        LevelTooLow    = 2,
        LevelTooHigh   = 3,
        FactionLocked  = 4,
        DayNightLocked = 5,
        QuestLocked    = 6,
        PartyRequired  = 7,
        InstanceFull   = 8,

    uint64_t instanceTicket;     // ticket d'entrée (si Success)

};
```

##### C.3 — Handler master

`src/masterd/handlers/instance/InstanceEnterHandler.h/.cpp` :

1. Valide la session client (ConnectionSessionMap → accountId).
2. Récupère le `Character` depuis DB (level, faction, completed_quests).
3. Charge `dungeons[dungeonId]` depuis DB.
4. Applique les checks (level, faction, quête, jour/nuit via M100.25 si dispo).
5. Si tous les checks passent : pour ce ticket, **retourne `NotImplemented`** avec un message « Not yet implemented — dungeon runtime in future ticket ». Le hook réseau est prêt, l'allocation d'instance suivra.

##### C.5 — Bridge éditeur → DB

L'éditeur écrit `instances/dungeons.bin` en local. Ce flux est volontairement manuel pour éviter de coupler l'éditeur à la DB.
