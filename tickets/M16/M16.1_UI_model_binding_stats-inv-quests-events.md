# M16.1 — UI model binding: stats/inv/quests/events

## Objectif
Créer UI Model (données) et binding depuis messages réseau.

## Dépendances
- M13.3

## Livrables
- UIModel struct
- Handlers messages -> UIModel
- Observer callbacks

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

## Spécification technique
- Séparer UI logique vs style; model thread-safe (main thread).

## Étapes d’implémentation
1. Définir UIModel (player stats, inventory, quest list, events).
2. Brancher handlers réseau pour snapshots/deltas.
3. Notifier vues (signals) ou pull-based.
4. Debug view dumps.

## Critères de validation (DoD)
- [ ] Changements réseau reflétés instantanément dans le model.

## Notes / pièges à éviter
- Éviter allocations massives à chaque delta; réutiliser buffers.
