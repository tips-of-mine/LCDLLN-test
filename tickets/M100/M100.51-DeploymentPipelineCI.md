# M100.51 — Deployment Pipeline CI (Phase 12 — Accessibilité — clôture)

## Résumé

**Dernier ticket de la Phase 12 et clôture de l'accessibilité éditeur.** Automatise l'intégralité du workflow de déploiement d'une zone vers un serveur de staging, qui était manuel et fastidieux après M100.44 (8 étapes en ligne de commande). Le mappeur n'a plus qu'à `git push` une branche `zone/<nom>` ; un workflow GitHub Actions prend le relais :

1. Détecte les zones modifiées dans le bundle.
2. Valide chacune via `ZoneValidator` (M100.48) — échec si Error détectée.
3. Lance `lcdlln_vmap_extractor` (M100.44) pour re-bake les `.vmap` tiles.
4. Synchronise `dungeons.bin` vers la DB via `dungeon_db_sync` (M100.43).
5. Met à jour `zone_content_versions` avec le nouveau hash.
6. Déploie les artefacts (vmap, mesh_inserts, terrain, splat) vers un serveur de staging par rsync.
7. Effectue un cold restart du shard staging et vérifie le boot OK.
8. Poste un commentaire sur la PR avec :
   - Le lien `staging:<port>` pour tester la zone live.
   - Le rapport de validation (warnings et hints).
   - Le delta de taille des artefacts.
   - Une capture rendue offline (option).

Le mappeur reçoit ainsi un **lien testable** quelques minutes après son push, sans toucher à la ligne de commande, sans connaissance des outils internes, sans accès SSH au serveur. C'est ce qui rend l'éditeur **réellement utilisable par des contributeurs externes**.

Le ticket livre également :
- Un workflow `build-and-deploy-zone.yml` GitHub Actions.
- Un workflow `validate-zone.yml` plus léger qui s'exécute sur toute PR touchant des zones (validation seule, pas de déploiement).
- Une infrastructure de **rollback automatique** : si le shard ne boot pas en 60 s, restauration des `.vmap` précédents.
- Une **doc de runbook** `docs/CI_ZONE_DEPLOYMENT.md` qui explique le pipeline aux contributeurs.

**Pas de nouvelle application interactive.** Le ticket ajoute un seul outil CLI offline mineur, `zone_validator_cli` (wrapper en ligne de commande autour de `ZoneValidator` de M100.48), sur le même pattern que `dungeon_db_sync` (M100.43) et `lcdlln_vmap_extractor` (M100.44). L'éditeur reste un binaire unique (`world_editor_app`). Le reste du ticket = workflows YAML, scripts shell, et configuration runtime serveur staging.

## Dépendances

- **M100.43** — `dungeon_db_sync` outil offline.
- **M100.44** — `lcdlln_vmap_extractor`, format `.vmap` v(N+1).
- **M100.48** — `ZoneValidator` (mode CLI).
- **SERVER-CORE.15** (existant) — Docker deployment infrastructure (STAB.15 cf. userMemories).
- **Phase 3** — Master + shard runtime.

## Contexte critique à respecter

### 1. Pipeline TOTALEMENT automatisable

Le pipeline doit fonctionner sans intervention humaine entre `git push` et `staging:port disponible`. Aucune étape n'attend une approbation manuelle.

Exception : si la PR contient un changement de format binaire détecté (bump de `kVmapFormatVersion`, etc.), le pipeline marque la PR comme « requires manual approval » et n'auto-déploie pas. Sécurité pour éviter qu'un changement de format casse staging.

### 2. Staging séparé de production

Le pipeline déploie sur un environnement **staging** isolé :
- Serveur dédié `staging.lcdlln.example` (à provisionner par Hubert).
- Base de données séparée `dungeons_staging`.
- Aucun joueur réel ne s'y connecte.
- Tests automatiques en post-déploiement (smoke tests).

Le déploiement vers production reste **manuel** et hors scope de ce ticket. Idem pour les zones officielles du jeu : elles passent par un autre process (vérification éditoriale, etc.).

### 3. Sécurité credentials

Le serveur staging stocke les artefacts pushed via SSH. Les secrets GitHub Actions :
- `STAGING_SSH_KEY` : clé privée d'un compte de déploiement.
- `STAGING_HOST` : `staging.lcdlln.example`.
- `STAGING_DB_PASSWORD` : pour `dungeon_db_sync`.

Tous via `secrets:` GitHub, jamais en clair dans les YAML.

### 4. Rollback automatique

Si après déploiement :
- Le shard ne démarre pas en 60 s.
- Les smoke tests post-deploy échouent.
- Le pipeline détecte une erreur réseau.

→ **Rollback automatique** :
1. Restauration des `.vmap` précédents (gardés dans `/var/lcdlln/staging/vmap_tiles.previous/`).
2. Restauration de `zone_content_versions` à la valeur précédente.
3. Cold restart shard avec les anciens artefacts.
4. Commentaire sur la PR : « 🛑 Déploiement échoué, rollback automatique effectué. Voir logs. »

### 5. Limites de fréquence

Pour éviter un mappeur qui spamme des pushes :
- Maximum 1 déploiement par PR toutes les 5 minutes.
- Maximum 20 déploiements par jour par PR.
- Au-delà : skip avec message « Limite atteinte, attendre N minutes ».

### 6. Idempotence

Si le pipeline est relancé sur le même commit (re-run), le résultat doit être identique. Pas de race condition, pas de duplication d'inserts DB.

### 7. Arbo server-core-style

- `.github/workflows/build-and-deploy-zone.yml` — workflow principal.
- `.github/workflows/validate-zone.yml` — validation seule.
- `tools/zone_deployment/` — scripts shell helpers.
- `docs/CI_ZONE_DEPLOYMENT.md` — doc.

#### Partie A — Workflow validate-zone.yml

Léger, s'exécute sur **toute PR** qui touche `game/data/zones/**` ou `game/data/editor/**` :

```yaml
name: Validate Zone

on:
  pull_request:
    paths:
      - 'game/data/zones/**'
      - 'game/data/editor/zone_presets/**'
      - 'game/data/editor/tool_presets/**'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4.2.2
      - name: Build zone_validator CLI
        run: cmake --build . --target zone_validator_cli
      - name: Detect changed zones
        id: changed
        run: ./tools/zone_deployment/detect_changed_zones.sh
      - name: Validate each changed zone
        run: |
          for zone in ${{ steps.changed.outputs.zones }}; do
            ./build/zone_validator_cli --zone-bundle "$zone" --format json
          done
      - name: Post results as PR comment
        if: always()
        uses: actions/github-script@v7
        with:
          script: |
            // ... post results
```

##### A.1 — Mode CLI de `ZoneValidator`

Nouvel exécutable mineur `zone_validator_cli` (binaire fin, link `engine_core`) :

```
Usage: zone_validator_cli --zone-bundle <path> [--format {text|json}] [--min-severity {hint|warning|error}]

Output (JSON) :
{
  "zoneId": "rocky_coast_01",
  "validation": {
    "errorCount": 0,
    "warningCount": 3,
    "hintCount": 5,
    "wallTimeMillis": 1242,
    "issues": [ /* ValidationIssue[] */ ]
  },
  "exitCode": 0
}
```

Exit code :
- 0 : pas d'Error (warnings/hints OK).
- 1 : Error détectée.
- 2 : Erreur fatale (zone introuvable, bundle corrompu).

#### Partie B — Workflow build-and-deploy-zone.yml

Workflow principal, plus lourd, s'exécute sur **push** vers une branche `zone/*` :

```yaml
name: Build & Deploy Zone (Staging)

on:
  push:
    branches:
      - 'zone/**'

concurrency:
  group: deploy-staging-${{ github.ref }}
  cancel-in-progress: true

jobs:
  rate-limit-check:
    runs-on: ubuntu-latest
    steps:
      - name: Check rate limit (5 min between deploys)
        run: ./tools/zone_deployment/rate_limit_check.sh

  validate-and-build:
    needs: rate-limit-check
    runs-on: ubuntu-latest
    outputs:
      changed-zones: ${{ steps.detect.outputs.zones }}
    steps:
      - uses: actions/checkout@v4.2.2
      - name: Build all required tools
        run: |
          cmake --preset linux-x64-release
          cmake --build --preset linux-x64-release --target \
            zone_validator_cli lcdlln_vmap_extractor dungeon_db_sync
      - name: Detect changed zones
        id: detect
        run: ./tools/zone_deployment/detect_changed_zones.sh
      - name: Validate each zone (must pass)
        run: |
          for zone in ${{ steps.detect.outputs.zones }}; do
            ./build/zone_validator_cli --zone-bundle "$zone" --min-severity error
          done
      - name: Bake vmaps
        run: |
          for zone in ${{ steps.detect.outputs.zones }}; do
            ./build/lcdlln_vmap_extractor --zone-bundle "$zone" --out "$zone/vmap_tiles/" --threads 8
          done
      - name: Upload artifacts
        uses: actions/upload-artifact@v4.6.2
        with:
          name: zone-artifacts
          path: |
            game/data/zones/**/*.bin
            game/data/zones/**/vmap_tiles/*.vmap
          retention-days: 7

  deploy-staging:
    needs: validate-and-build
    runs-on: ubuntu-latest
    environment: staging
    steps:
      - uses: actions/download-artifact@v4.6.2
        with: { name: zone-artifacts, path: ./artifacts }
      - name: Backup current staging vmaps
        run: |
          ssh -i "$SSH_KEY" "$STAGING_HOST" \
            "cp -r /var/lcdlln/staging/vmap_tiles /var/lcdlln/staging/vmap_tiles.previous"
      - name: Rsync artifacts to staging
        run: |
          rsync -avz -e "ssh -i $SSH_KEY" ./artifacts/ \
            "$STAGING_HOST:/var/lcdlln/staging/"
      - name: Sync dungeons DB
        run: |
          ssh -i "$SSH_KEY" "$STAGING_HOST" \
            "/opt/lcdlln/dungeon_db_sync \
              --in /var/lcdlln/staging/game/data/zones/**/instances/dungeons.bin \
              --db mysql://lcdlln:$STAGING_DB_PASSWORD@localhost/dungeons_staging"
      - name: Update zone_content_versions
        run: |
          # SQL UPDATE pour chaque zone modifiée
          ./tools/zone_deployment/update_content_versions.sh
      - name: Cold restart shard staging
        run: |
          ssh -i "$SSH_KEY" "$STAGING_HOST" \
            "sudo systemctl restart lcdlln-shard-staging"
      - name: Wait for boot OK (max 60s)
        id: boot-check
        run: |
          ./tools/zone_deployment/wait_for_boot.sh \
            --host "$STAGING_HOST" --port 5101 --timeout 60
      - name: Smoke tests
        if: success()
        run: |
          ./tools/zone_deployment/run_smoke_tests.sh \
            --host "$STAGING_HOST" --port 5101
      - name: Rollback on failure
        if: failure()
        run: |
          ./tools/zone_deployment/rollback.sh --host "$STAGING_HOST"

  notify:
    needs: deploy-staging
    runs-on: ubuntu-latest
    if: always()
    steps:
      - name: Post PR comment with result
        uses: actions/github-script@v7
        with:
          script: |
            // Détaillé en C
```

#### Partie C — Scripts helpers

##### C.1 — `detect_changed_zones.sh`

Compare le commit précédent avec le commit courant via `git diff`, retourne la liste des zones touchées. Bash simple :

```bash
#!/usr/bin/env bash
set -euo pipefail
CHANGED_FILES=$(git diff --name-only "$GITHUB_EVENT_BEFORE" "$GITHUB_SHA" -- 'game/data/zones/**')
ZONES=$(echo "$CHANGED_FILES" | awk -F'/' '{print $4}' | sort -u)
echo "zones=$ZONES" >> "$GITHUB_OUTPUT"
```

##### C.2 — `rate_limit_check.sh`

Lit `last_deploy.json` dans le repo (committed dans `.github/state/`) et vérifie si `(now - last_deploy_time) > 5min`. Si non, exit 1. Met à jour le fichier après chaque deploy réussi (via commit auto par le workflow).

##### C.3 — `wait_for_boot.sh`

Polling TCP sur l'endpoint shard staging toutes les 2 s, jusqu'à `--timeout`. Exit 0 si connection OK avant timeout, sinon exit 1.

##### C.4 — `run_smoke_tests.sh`

Suite de smoke tests automatiques :
- Login d'un compte test.
- Spawn dans la zone modifiée.
- Vérification visuelle (rendu d'une frame, comparaison pixel avec référence — optionnel).
- Vérification que `GetHeight()` répond.
- Vérification que les mesh inserts sont visibles côté server (raycast contre un mesh insert connu).

Exit 0 si tout OK, sinon exit 1 avec rapport détaillé.

##### C.5 — `rollback.sh`

Restaure les `.vmap` depuis `vmap_tiles.previous/`, restaure `zone_content_versions`, redémarre le shard.

##### C.6 — `update_content_versions.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail
for zone in "$@"; do
  HASH=$(sha256sum "game/data/zones/$zone/instances/mesh_inserts.bin" | awk '{print $1}')
  mysql -h "$STAGING_HOST" -u lcdlln -p"$STAGING_DB_PASSWORD" dungeons_staging \
    -e "UPDATE zone_content_versions SET mesh_inserts_hash='$HASH', updated_at=NOW() WHERE zone_id='$zone';"
done
```
