- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Endianness: **little-endian**
- Max packet size: **16 KB** (hard limit)
- Header: **fixed-size**, strictement packed, no padding
  - `uint16 size` (taille totale paquet, header inclus)
  - `uint16 opcode`
  - `uint16 flags`
  - `uint32 request_id`
  - `uint64 session_id`
- Strings: `uint16 length` + bytes UTF-8 (sans null terminator)
- Arrays: `uint16 count` + éléments contigus
- Compression: none (v1)
- Encryption: **TLS only**
- Messaging: request/response + server push
- `request_id`: utilisé pour matching; `0` réservé aux messages sans réponse attendue (server push)
- Errors: paquet `ERROR` unique (code numérique + message optionnel)

## Étapes d'implémentation
1. Rédiger `protocol_v1.md` (header, règles encodage, limites).
2. Définir opcodes v1 et leurs payloads (structure binaire).
3. Définir error codes v1 + règles de réaction (ERROR vs disconnect).
4. Ajouter 2 exemples complets (AUTH_REQUEST, ERROR par exemple) en hex.

## Critères de validation (DoD)
- [ ] La doc suffit à implémenter encode/decode sans ambiguïté.
- [ ] Toutes les tailles, types et limites sont explicitement définies.
- [ ] Tables opcodes + error codes stables et versionnées.

## Notes / pièges à éviter
- Ne pas inclure compression/delta v2 dans v1.
- Ne pas compter sur `struct` packé en mémoire pour l'implémentation (risque UB); documenter mais écrire champ par champ.
