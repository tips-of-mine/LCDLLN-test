# M21.4 — MySQL users & permissions: master vs shard

**Status:** Closed

## Objectif
Mettre en place deux utilisateurs MySQL distincts: master (RW all) et shard (RW limité aux tables gameplay), et documenter l'installation.

## Dépendances
- M21.1

## Livrables
- `db/users.sql`:
  - create user master_user
  - create user shard_user
  - GRANT appropriés
- `docs/db_permissions.md`

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Master: full read/write all tables.
- Shard: restricted to gameplay-related tables (characters, inventory, etc.) — pas d'accès accounts/sessions.
- Appliquer le principe du moindre privilège.

## Étapes d'implémentation
1. Définir liste tables "auth" vs "gameplay".
2. Écrire grants.
3. Documenter procédure.

## Critères de validation (DoD)
- [ ] shard_user ne peut pas SELECT/UPDATE accounts.
- [ ] master_user peut migrer.
