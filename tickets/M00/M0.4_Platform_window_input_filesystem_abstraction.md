# M0.4 — Platform: window, input, filesystem abstraction

## Objectif
Abstraction plateforme : création fenêtre, input (clavier/souris), chemins, lecture fichiers, time monotone (si besoin).

## Dépendances
- M0.1

## Livrables
- `/engine/platform/Window.*` (GLFW recommandé) + events resize/close
- `/engine/platform/Input.*` (WASD, mouse delta, key states)
- `/engine/platform/FileSystem.*` (read file, exists, list, path join)


## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

## Spécification technique
- Fenêtre: création, resize callback, fullscreen toggle (option).
- Input: état touches, mouse delta, capture/release curseur.
- FS: lecture binaire/texte, chemins relatifs à `paths.content`.

## Étapes d’implémentation
1. Intégrer GLFW (ou Win32 si Windows-only).
2. Implémenter Window::PollEvents + callbacks vers engine.
3. Implémenter Input: update par frame + mouse look.
4. Implémenter FileSystem: ReadAllBytes/ReadAllText.
5. Smoke test: fenêtre + caméra libre.

## Critères de validation (DoD)
- [ ] Fenêtre s'ouvre, se ferme proprement, resize détecté.
- [ ] WASD + souris fonctionne dans loop.
- [ ] Lecture fichier config/asset via FS OK.

## Notes / pièges à éviter
- Isoler GLFW dans platform pour pouvoir swap plus tard.
- Ne pas bloquer sur I/O dans le thread render.
