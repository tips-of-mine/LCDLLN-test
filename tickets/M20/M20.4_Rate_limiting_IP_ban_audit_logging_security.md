# M20.4 — Security hardening: rate limiting, IP ban, audit logging

## Objectif
Implémenter protections auth v1: rate limiting, blacklist IP temporaire, audit logging (sécurité).

## Dépendances
- M19.9 (transport hardening)
- M20.3 (sessions) pour corréler événements
- M00.1 (logging)

## Livrables
- Rate limiting (token bucket) par IP:
  - AUTH attempts / minute
  - REGISTER attempts / hour
- IP ban temporaire après X échecs:
  - ban duration configurable
- Audit log dédié (fichier séparé ou logger séparé):
  - login success/fail
  - register success/fail
  - bans/unbans
  - session created/closed
- Counters exportables (pré M23)

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Structures en mémoire + purge périodique.
- Les bans doivent s’appliquer avant traitement coûteux (avant DB si possible).
- Policy de log: ne jamais logger passwords/hashes.

## Étapes d’implémentation
1. Token bucket par IP + endpoints auth/register.
2. Compter échecs + déclencher ban.
3. Logger “security_audit” séparé.
4. Ajouter tests unitaires sur limiter + ban.

## Critères de validation (DoD)
- [ ] Flood auth -> limité, serveur stable.
- [ ] IP ban déclenché et expiré correctement.
- [ ] Audit log contient événements essentiels.
