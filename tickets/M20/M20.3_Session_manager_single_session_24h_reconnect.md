# M20.3 — Session manager: single-session, 24h expiry, reconnection window

## Objectif
Créer un gestionnaire de sessions global: une seule session active par compte, session_id 64-bit, expiry 24h, reconnexion courte.

## Dépendances
- M19.7 (request_id dispatcher)
- M19.11 (counters)
- M21 (DB) pour persistance sessions si nécessaire (sinon mémoire v1)

## Livrables
- Session state machine:
  - created → authenticated → active → expired/closed
- `session_id` 64-bit généré côté serveur (CSPRNG ou monotonic + entropy)
- Règles:
  - 1 session active/account (duplicate login => refuse ou kick ancien selon policy v1)
  - expiry: logout, heartbeat timeout, ou 24h max
  - reconnection window: N secondes (config)
- API:
  - CreateSession(account_id) -> session_id
  - Validate(session_id)
  - Touch/Heartbeat(session_id)
  - Close(session_id, reason)

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Stocker: account_id, session_id, created_at, last_seen, expires_at, state.
- Heartbeat interval/timeout aligné server_config.ini.
- Clock monotonic pour timeouts.

## Étapes d’implémentation
1. Définir struct session + tables de lookup (by session_id, by account_id).
2. Implémenter règles single-session.
3. Implémenter expiry 24h + heartbeat timeout.
4. Implémenter reconnexion (resume si dans fenêtre).
5. Tests unitaires sur transitions d’état.

## Critères de validation (DoD)
- [ ] Duplicate login géré conformément policy.
- [ ] Expiry 24h enforce.
- [ ] Resume possible dans fenêtre.
