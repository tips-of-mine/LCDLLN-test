# M20.2 — Argon2 double-hash: protocol + implementation

## Objectif
Implémenter le double hash Argon2: client_hash côté client, final_hash côté serveur, et stocker uniquement `final_hash` + `server_salt` en DB.

## Dépendances
- M19.3 (client net)
- M19.4/M19.5 (server net + TLS)
- libargon2 disponible côté client+serveur

## Livrables
- Implémentation côté client:
  - `client_hash = Argon2(password, client_salt)`
  - envoyer uniquement `client_hash` (TLS)
- Implémentation côté serveur:
  - `final_hash = Argon2(client_hash, server_salt)`
  - stockage `final_hash`, `server_salt`
- Paramètres Argon2 v1 (configurable, documentés)
- Tests unitaires hashing wrapper

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Le client ne transmet jamais le plaintext password.
- Le serveur ne stocke jamais le plaintext password.
- Le `server_salt` est unique par compte (secure random).
- Les paramètres Argon2 doivent être stockés (ou versionnés) pour upgrades futurs.

## Étapes d’implémentation
1. Wrapper argon2 côté client+serveur.
2. Génération salts (CSPRNG).
3. Format d’encodage du hash (string standard argon2id recommandé).
4. Tests: verify(pass) sur serveur.
5. Documenter paramètres.

## Critères de validation (DoD)
- [ ] Aucun chemin ne transmet/stock le plaintext password.
- [ ] Verify fonctionne sur hash stocké.
