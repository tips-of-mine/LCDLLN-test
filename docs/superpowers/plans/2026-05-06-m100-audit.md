# M100 — Audit & Gap Analysis Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produire le document `docs/superpowers/audits/2026-05-06-m100-gap-analysis.md` qui audite les 34 tickets M100 (`tickets/M100/`) au niveau fichier, contre le code existant, avec verdict (done / partiel / vide), effort estimé (XS–XL) et flags transverses (pivot M100.11, redéploiement serveur, tests round-trip).

**Architecture :** Dispatch parallèle de 10 sous-agents `Explore` (un par phase M100) en lecture seule, puis consolidation en main session avec tableau récap, flags transverses et bloc « risques détectés ». Aucun code C++ produit. Le plan est *strictement* un livrable documentaire — la sortie alimente les 10 plans de build qui seront écrits ensuite (un par phase, via une seconde invocation de `superpowers:writing-plans`).

**Tech Stack :** Markdown, git, Agent tool (subagent_type=Explore), Bash, Read, Write, Edit.

**Périmètre verrouillé par le design :** [docs/superpowers/specs/2026-05-06-m100-execution-design.md](../specs/2026-05-06-m100-execution-design.md) — décisions §2 questions 1–6.

**Pourquoi un plan dédié à l'audit ?** Le design §10 prévoit l'audit comme étape 1 du plan général. Mais les étapes 2–11 (les 10 PRs de build) dépendent par construction des chiffres verdict/effort/risques que l'audit produit. Tant que l'audit n'a pas livré, il est impossible d'écrire les tâches de build au niveau code-level requis par `writing-plans`. Donc : **ce plan ne couvre que l'audit**. À la fin, la tâche 6 invoque à nouveau `writing-plans` pour produire les plans de build, un par phase, en consommant l'audit comme entrée.
