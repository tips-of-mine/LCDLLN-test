## Contenu du ticket

# M00.1.2 — Log boot complet : fichier horodaté + instrumentation séquence de démarrage

## Objectif
Actuellement, même avec `-log`, aucune sortie n'est visible car `Log::Init()` est
appelé mais **aucune ligne `LOG_*` ne trace la séquence de boot** dans
`Engine::Engine()`. De plus, le fichier `engine.log` est écrasé à chaque lancement,
rendant impossible la comparaison de runs successifs.

Ce ticket corrige les deux points :
1. **Fichier log horodaté** : nommé `lcdlln.exe-YYYYMMDD-HHMMSS.log` pour conserver
   l'historique de chaque lancement.
2. **Instrumentation complète** : ajout de `LOG_INFO` / `LOG_WARN` / `LOG_ERROR` à
   chaque étape critique du constructeur `Engine::Engine()` et des sous-systèmes
   qu'il initialise.

## Dépendances
- M00.1.1 (Log::Init via flag `-log` déjà implémenté)

## Livrables
- `engine/Engine.cpp` — nom de fichier log horodaté + `LOG_*` à chaque étape du boot
- `engine/core/Log.cpp` / `engine/core/Log.h` — si modification du chemin de fichier
  nécessaire (ajout d'un helper de génération du nom horodaté)

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).

## Spécification technique

### 1) Nom de fichier horodaté

Générer le nom **avant** `Log::Init()`, en utilisant `std::chrono` et `std::strftime` :

```cpp
// Dans Engine::Engine(), avant Log::Init()
{
    const auto now   = std::chrono::system_clock::now();
    const std::time_t t = std::chrono::system_clock::to_time_t(now);
    std::tm tm{};
#if defined(_WIN32)
    localtime_s(&tm, &t);
#else
    localtime_r(&t, &tm);
#endif
    char buf[64]{};
    std::strftime(buf, sizeof(buf), "lcdlln.exe-%Y%m%d-%H%M%S.log", &tm);
    logSettings.filePath = buf;  // ex: "lcdlln.exe-20250308-143022.log"
}
```

Le fichier est créé dans le répertoire courant (à côté de l'exe), sans sous-dossier.

### 2) Instrumentation de la séquence de boot dans `Engine::Engine()`

Ajouter un `LOG_INFO` ou `LOG_WARN`/`LOG_ERROR` **à chaque étape** :

```
[Boot] Log initialized  (console=on|off, file=lcdlln.exe-....log)
[Boot] Config loaded    (vsync=on|off, fixed_dt=X)
[Boot] Window::Create   OK  /  FATAL
[Boot] glfwInit         OK  /  WARN failed
[Boot] glfwCreateWindow OK  /  WARN returned null
[Boot] VkInstance::Create       OK  /  WARN failed
[Boot] VkInstance::CreateSurface OK  /  WARN failed
[Boot] VkDeviceContext::Create  OK (GPU: <deviceName>)  /  WARN failed
[Boot] VkSwapchain::Create      OK (extent=WxH, images=N)  /  WARN failed
[Boot] FrameSync::Init  OK
[Boot] FrameArena init  OK
[Boot] ShaderCompiler   OK  /  WARN glslangValidator not found
[Boot] DeferredPipeline init    OK  /  WARN
[Boot] JobSystem init   OK (workers=N)
[Boot] Engine boot COMPLETE
```

Chaque LOG doit être émis **immédiatement après** l'opération, avant la suivante,
afin qu'en cas de crash le dernier log visible indique l'étape bloquante.

### 3) Règle `flushAlways = true`
Doit rester `true` pour que chaque ligne soit physiquement écrite avant le prochain
appel — garantit que le fichier est lisible même après un `abort()` ou un crash OS.

## Étapes d'implémentation

1. **`engine/Engine.cpp`** — constructeur :
   - Générer `logSettings.filePath` horodaté.
   - Conserver `flushAlways = true`.
   - Ajouter `LOG_INFO(Core, "[Boot] Log initialized ...")` juste après `Log::Init()`.
   - Ajouter `LOG_INFO` / `LOG_WARN` / `LOG_ERROR` après **chaque** étape existante
     (Config, Window, glfwInit, glfwCreateWindow, VkInstance, VkSurface,
     VkDeviceContext, VkSwapchain, FrameSync, ShaderCompiler, DeferredPipeline,
     JobSystem).
   - Terminer par `LOG_INFO(Core, "[Boot] Engine boot COMPLETE")`.

2. **Pas de modification** de l'architecture ou d'autres fichiers sauf si la
   génération du nom horodaté nécessite un helper dans `Log.h`/`Log.cpp`
   (dans ce cas, ajouter `static std::string MakeTimestampedFilename(std::string_view prefix)`).

## Notes / pièges à éviter
- Ne **pas** utiliser `std::format` pour le nom de fichier : `strftime` est plus
  simple et sans dépendances supplémentaires.
- Ne **pas** supprimer l'ancien `engine.log` — le nouveau nom horodaté le remplace
  de facto ; si `log.file` est présent dans `config.json`, le nom horodaté
  **prend priorité** quand `-log` est actif.
- Les `LOG_WARN` sur les échecs Vulkan non-fatals (surface, device) doivent rester
  des WARN et non des FATAL pour laisser le moteur tenter de continuer et logger
  la suite.
