# M13.1 — Server core: tick loop + connections + snapshots skeleton

## Objectif
Serveur headless avec tick fixe (20/30Hz), gestion connexions, envoi snapshots vides.

## Dépendances
- M0.3
- M0.5

## Livrables
- Server app
- Network transport (UDP/ENet/quic?)
- Tick scheduler


## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

## Spécification technique
- Serveur authoritative; client envoie inputs; snapshots à 10-20Hz.

## Étapes d’implémentation
1. Choisir transport (UDP + reliability layer ou ENet).
2. Implémenter accept/handshake, clientId.
3. Loop tick: process inputs -> sim -> build snapshot.
4. Client: receive snapshot + stats.

## Critères de validation (DoD)
- [ ] 2 clients connectent et reçoivent ticks/snapshots réguliers.

## Notes / pièges à éviter
- Séparer protocole des données de jeu (serialisation).
