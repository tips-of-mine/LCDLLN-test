# M22.1 — Master: shard registry model + state

**Status:** Closed

## Objectif
Définir le modèle de shard (state machine) et le registre côté Master (source of truth) incluant capacité, load, disponibilité.

## Dépendances
- M19 (transport)
- M21.1 (table shards)

## Livrables
- State machine shard:
  - registering → online → degraded → offline
- Champs:
  - shard_id
  - max_capacity
  - current_load
  - last_heartbeat
  - region/endpoint (si applicable)
- API interne: RegisterShard, UpdateHeartbeat, MarkDown

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Master décide uniquement (source of truth).
- Timeout heartbeat => offline.

## Étapes d'implémentation
1. Définir struct shard state.
2. Implémenter registre en mémoire + persistence optionnelle.
3. Policy de sélection de shard (placeholder pour M22.4).

## Critères de validation (DoD)
- [ ] Un shard peut apparaître "online" après register+heartbeat.
