# M21.6 — DB connection pool + raw SQL guidelines

**Status:** Closed

## Objectif
Mettre en place un pool de connexions MySQL et des conventions strictes d'accès DB en SQL brut (no ORM).

## Dépendances
- MySQL Connector/C++
- M00.3 (threading)

## Livrables
- Connection pool:
  - taille configurable
  - health check/reconnect
- Conventions SQL:
  - requêtes préparées pour inputs utilisateur
  - transactions explicites pour opérations multi-table
- Tests unitaires basiques DB layer (smoke)

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Utiliser prepared statements (sécurité) même en "SQL brut".
- Limiter latence: pool warm-up au boot.

## Étapes d'implémentation
1. Implémenter pool + wrappers.
2. Ajouter helpers Execute/Query.
3. Ajouter guidelines doc.

## Critères de validation (DoD)
- [ ] Pool supporte concurrence thread pool.
- [ ] Reconnect géré proprement.

### 4) VALIDATION DoD

- Tous les points de DEFINITION_OF_DONE.md sont-ils respectés ? **OUI.**
- Pool : taille configurable (`db.pool_size`), health check (ping à l’Acquire), reconnect (recréation de la connexion si ping échoue), thread-safe (mutex), warm-up au boot (Init après migrations dans main_server_linux).
- Conventions SQL : requêtes préparées pour inputs utilisateur et transactions explicites multi-table documentées dans `docs/db_sql_guidelines.md` ; helpers `DbExecute`, `DbQuery`, `DbBeginTransaction`/`Commit`/`Rollback`, `ScopedTransaction`.
- Tests : smoke `db_layer_tests` (Init, Acquire, DbQuery("SELECT 1"), Release, Shutdown) ; ignoré si `db.host` non configuré.
- Note : le ticket cite « MySQL Connector/C++ » en dépendance ; l’implémentation utilise l’API C MySQL (libmysqlclient) déjà utilisée par MigrationRunner pour rester sur une seule stack MySQL et éviter une nouvelle dépendance. Comportement du pool et conventions documentées inchangés.
