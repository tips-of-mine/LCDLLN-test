
**Objectif**
Prédire mouvement client-side pour réactivité, buffer inputs pour envoyer au serveur.

**Dépendances**
- M26.1 (character controller)
- M13.1 (networking)

**Livrables**
- Client-side movement simulation (immediate response)
- Input buffer (queue commands)
- Send inputs to server (tick-based)

**Structure & chemins (verrouillé)**
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

**Spécification technique**
- Client: simulate movement immédiatement (no wait server)
- Buffer inputs avec tick number
- Send InputCommand(tick, keys, mouseDelta) à 20-30 Hz
- Server: replay inputs, simulate authoritative

**Étapes d'implémentation**
1. Client: input -> update local character position (optimistic)
2. Store input dans buffer avec tick#
3. Send inputs batch au server chaque network tick
4. Server: receive inputs, replay, compute authoritative state
5. Server: send snapshot (position, velocity, tick) back
6. Client: reconcile (next section)

**Critères de validation (DoD)**
- [ ] Mouvement local instantané (no latency feel)
- [ ] Server reçoit inputs et simule
- [ ] Snapshots retournent au client

**Notes / pièges à éviter**
- Fixed timestep important (client et serveur même dt)
- Input ID pour tracking reconciliation
