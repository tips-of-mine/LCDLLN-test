# M19.8 — Centralized error codes + ERROR packet handling

## Objectif
Standardiser les erreurs réseau v1 avec codes numériques centralisés et un paquet `ERROR` unique, aligné sur la spec M19.1.

## Dépendances
- M19.1
- M19.2
- M00.1

## Livrables
- Enum `NetErrorCode` (stable, doc)
- Packet `ERROR`:
  - `uint32 error_code`
  - `string message` (optionnel)
  - `uint32 request_id_ref` (optionnel)
- Helpers build/parse ERROR
- Policy serveur:
  - quand renvoyer ERROR vs disconnect

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Codes stables (pas de réassign sans bump version doc).
- En prod, message optionnel doit éviter fuite d’infos internes.

## Étapes d’implémentation
1. Définir NetErrorCode + synchroniser doc.
2. Implémenter encode/decode ERROR.
3. Intégrer: unknown opcode, invalid packet, oversize.
4. Côté client: handler ERROR (log + hook UI future).

## Critères de validation (DoD)
- [ ] Unknown opcode → ERROR(UNKNOWN_OPCODE) ou policy.
- [ ] Invalid packet → ERROR(INVALID_PACKET) ou disconnect.
- [ ] Logs affichent code + contexte.

## Notes / pièges à éviter
- Ne pas spammer ERROR en boucle (couper après N occurrences).
