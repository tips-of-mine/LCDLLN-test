# M44.3_Load_balancing_multi-server_zones_+_sharding

## Objectif
Support multi-serveurs avec sharding zones, load balancing players.

## Dépendances
- M13.1
- M09.1

## Livrables
- Zone sharding: assign zones to servers
- Player routing: connect to correct server
- Inter-server communication (zone transitions)
- Load balancer (frontend)

## Structure & chemins (verrouillé)
- Code moteur : uniquement sous `/engine/...`
- Contenu (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- Outils offline : uniquement sous `/tools/...`
- Tous les chemins de contenu sont relatifs à `paths.content` (config.json).
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

## Spécification technique
- Architecture:
  - Login server (1)
  - Game servers (N) (each owns zones)
  - Database (shared)
- Sharding: zone_001 -> server_A, zone_002 -> server_B
- Routing: login server returns game server IP
- Transitions: player moves zone -> disconnect -> reconnect new server

## Étapes d'implémentation
1. Login server:
   - Authenticate player
   - Query player last zone
   - Lookup zone->server mapping
   - Return game server IP/port
2. Game server:
   - Register with login server (heartbeat)
   - Load assigned zones
   - Accept connections
3. Zone transition:
   - Save player state
   - Send redirect to client (new server IP)
   - Client disconnect, connect new server
   - Load player state
4. Load balancer (optional):
   - Distribute players across servers
   - Monitor server load (players, CPU)

## Notes / pièges à éviter
- Shared DB: connection pooling (avoid contention)
- Cross-server chat/party (message bus, Redis pub/sub)
