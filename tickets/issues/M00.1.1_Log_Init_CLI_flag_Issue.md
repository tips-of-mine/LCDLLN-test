## Contenu du ticket

# M00.1.1 — Log::Init conditionnel via flag CLI `-log`

## Objectif
`Log::Init()` n'est actuellement **jamais appelé** dans `Engine::Engine()`, ce qui
rend tout diagnostic impossible (aucun fichier `engine.log`, aucune sortie console).

Ce ticket corrige le problème en :
1. Appelant toujours `Log::Init()` au boot du moteur.
2. Activant la sortie **console** uniquement si l'argument `-log` est passé à
   l'exécutable — pour ne pas polluer le terminal en usage normal.
3. La sortie **fichier** (`engine.log`) reste toujours active (utile pour le
   support et le debug post-mortem).

## Dépendances
- M00.1 (Log, Config déjà implémentés)

## Livrables
- `engine/Engine.cpp` — ajout de `Log::Init()` en toute première instruction du
  constructeur, avec détection du flag `-log` dans `argv`.

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

## Spécification technique

### Détection du flag `-log`
Scanner `argv` **avant** la construction de `Config` (Config n'est pas encore
disponible à ce stade) :

```cpp
bool logToConsole = false;
for (int i = 1; i < argc; ++i)
{
    if (argv[i] && std::string_view(argv[i]) == "-log")
    {
        logToConsole = true;
        break;
    }
}
```

### Initialisation du logger
```cpp
engine::core::LogSettings logSettings;
logSettings.filePath    = "engine.log";   // toujours actif
logSettings.console     = logToConsole;   // actif seulement avec -log
logSettings.flushAlways = true;           // critique : évite toute perte en cas de crash
// Niveau par défaut Info ; sera surchargeable via config une fois disponible
logSettings.level = engine::core::LogLevel::Info;
engine::core::Log::Init(logSettings);
LOG_INFO(Core, "Log initialized (console={})", logToConsole ? "on" : "off");
```

Cet appel doit être la **toute première instruction** du corps du constructeur
`Engine::Engine()`, avant `m_vsync`, avant `Window::Create`, avant `glfwInit`.

### Comportement attendu

| Commande                  | Console | Fichier `engine.log` |
|---------------------------|:-------:|:--------------------:|
| `lcdlln.exe`              | ✗       | ✓                    |
| `lcdlln.exe -log`         | ✓       | ✓                    |

### Niveau de log surchargeable via CLI
L'argument `--log.level=Debug` existant (géré par `Config::ApplyCli`) peut être
utilisé **après** l'init pour affiner le niveau. Ce ticket ne modifie pas ce
comportement.

## Étapes d'implémentation
1. Dans `engine/Engine.cpp`, ajouter la boucle de détection `-log` en début de
   constructeur.
2. Appeler `Log::Init(logSettings)` immédiatement après.
3. Ajouter `LOG_INFO(Core, "Engine boot start")` pour confirmer que le log
   fonctionne.
4. Ne toucher à aucun autre fichier.

## Critères de validation (DoD)
- [ ] Build Windows OK (configure + build) via presets existants
- [ ] `lcdlln.exe` seul → pas de sortie console, fichier `engine.log` créé avec
  les logs de boot
- [ ] `lcdlln.exe -log` → logs visibles dans le terminal ET dans `engine.log`
- [ ] Aucun dossier racine non autorisé créé
- [ ] Rapport final : fichiers modifiés + commandes + résultats + DoD

## Notes / pièges à éviter
- Ne pas appeler `Log::Init()` après `Config::Load()` : si le constructeur
  crashe avant, on perd les logs. L'init doit être inconditionnelle et
  immédiate.
- `flushAlways = true` est indispensable en Release pour garantir que le fichier
  est écrit avant un éventuel `abort()`.
- Le flag `-log` est distinct du système `--key=value` de `Config::ApplyCli` :
  il est scanné manuellement pour être disponible avant toute init.
