# M22.6 — Client flow: Auth → ServerList → Ticket → Connect Shard

## Objectif
Assembler le “vertical slice” réseau distribué: client s’auth sur Master, récupère server list, demande ticket, connecte au shard avec ticket.

## Dépendances
- M19 (TLS/protocol)
- M20 (auth/sessions)
- M22.4 (ticket)

## Livrables
- Flow complet (headless acceptable):
  - AUTH
  - SERVER_LIST
  - REQUEST_SHARD_TICKET
  - CONNECT SHARD (ticket validation)
- Logs de bout en bout + erreurs claires

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Le client ne doit jamais conserver un ticket après usage.
- Timeout et retries raisonnables (pas flood).

## Étapes d’implémentation
1. Implémenter client orchestration.
2. Implémenter endpoints shard pour handshake ticket.
3. Tests intégration end-to-end.

## Critères de validation (DoD)
- [ ] Flow complet fonctionne sur machine locale.
- [ ] Connexion shard refusée sans ticket.
