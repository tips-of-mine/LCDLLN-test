# M19.1 — Protocol v1: specification (source of truth)

## Objectif
Définir la spécification officielle du protocole réseau binaire v1 (header, encodage, opcodes initiaux, erreurs) et en faire la **source de vérité** unique pour client et serveur.

## Dépendances
- M00.1

## Livrables
- `tickets/docs/protocol_v1.md` (spécification)
- Table officielle des opcodes v1 (incluant au minimum: AUTH_*, REGISTER_*, SERVER_LIST_*, HEARTBEAT, ERROR)
- Table officielle des error codes v1 (centralisée)
- Exemples de paquets (au moins 2) avec dump hex + explication champ par champ

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Endianness: **little-endian**
- Max packet size: **16 KB** (hard limit)
- Header: **fixed-size**, strictement packed, no padding
  - `uint16 size` (taille totale paquet, header inclus)
  - `uint16 opcode`
  - `uint16 flags`
  - `uint32 request_id`
  - `uint64 session_id`
- Strings: `uint16 length` + bytes UTF-8 (sans null terminator)
- Arrays: `uint16 count` + éléments contigus
- Compression: none (v1)
- Encryption: **TLS only**
- Messaging: request/response + server push
- `request_id`: utilisé pour matching; `0` réservé aux messages sans réponse attendue (server push)
- Errors: paquet `ERROR` unique (code numérique + message optionnel)

## Étapes d’implémentation
1. Rédiger `protocol_v1.md` (header, règles encodage, limites).
2. Définir opcodes v1 et leurs payloads (structure binaire).
3. Définir error codes v1 + règles de réaction (ERROR vs disconnect).
4. Ajouter 2 exemples complets (AUTH_REQUEST, ERROR par exemple) en hex.

## Critères de validation (DoD)
- [ ] La doc suffit à implémenter encode/decode sans ambiguïté.
- [ ] Toutes les tailles, types et limites sont explicitement définies.
- [ ] Tables opcodes + error codes stables et versionnées.

## Notes / pièges à éviter
- Ne pas inclure compression/delta v2 dans v1.
- Ne pas compter sur `struct` packé en mémoire pour l’implémentation (risque UB); documenter mais écrire champ par champ.
