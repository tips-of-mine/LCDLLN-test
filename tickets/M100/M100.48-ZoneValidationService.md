# M100.48 — Zone Validation Service (Phase 12 — Accessibilité)

## Résumé

Service de **validation automatique** d'une zone en cours d'édition qui détecte les problèmes courants (trous heightmap, rivières orphelines, mesh inserts en collision, donjons à ID dupliqué, splat cumulé incohérent, chunks dirty non re-bakés, etc.), les liste par sévérité dans un panel dédié, et propose pour chacun un bouton **« Aller à »** qui navigue la caméra du viewport jusqu'à la zone du problème.

Le service tourne :
- **Sur demande** : bouton « Valider la zone » dans la toolbar et menu Fichier.
- **Automatiquement avant export** : modal bloquant si des `Error` sont détectées, warning non-bloquant pour les `Warning`/`Hint`.
- **En arrière-plan** (optionnel, défaut OFF) : revalidation incrémentale toutes les N modifications, pour un feedback live.

L'architecture est **extensible** : chaque catégorie de validation est une `IValidationRule` indépendante enregistrée dans un registre. Phase 13 et tickets futurs ajouteront de nouvelles règles (vegetation density trop élevée, panneaux sans texte localisé, etc.) sans modifier le service.

**Aucun nouvel exécutable.** Code dans `engine_core`, binaire éditeur unique.

## Dépendances

- **M100.1** — Bootstrap.
- **M100.5** — Heightmap (validation cellules).
- **M100.9** — Splat (validation cumulé).
- **M100.13** — Water (validation rivières/lacs).
- **M100.40-43** — Mesh inserts (validation AABB, ID).
- **M100.47** — Help system (lien depuis un problème vers la doc explicative).

## Contexte critique à respecter

### 1. Pas de validation runtime in-game

Cette validation est **éditeur-only**. Le runtime jeu ne valide pas en live ; il assume que les données ont été validées avant export. C'est conforme au modèle ticket-driven : tout problème de qualité contenu est détecté à la création, pas à l'exécution.

### 2. Sévérités tranchées

Trois niveaux strictement :

- **❌ Error** : empêche l'export. Le contenu est corrompu ou incomplet, le runtime échouerait (ex : référence vers un glTF manquant, donjon avec ID dupliqué).
- **⚠ Warning** : impacte le gameplay ou le visuel mais n'empêche pas l'export. Le mappeur doit voir et décider (ex : mesh insert qui flotte au-dessus du terrain après érosion).
- **💡 Hint** : suggestion de style ou de best practice. Aucun blocage (ex : zone sans aucun point d'intérêt, splat unique sur > 80 % de la surface).

Pas d'autres niveaux. Pas de seuils configurables — chaque règle décide.

### 3. Performance : validation incrémentale

Une validation complète peut être longue (10-30 s sur une grande zone). On évite donc de tout re-valider après chaque modification. Stratégie :

- Mode « Sur demande » : full pass.
- Mode « Background » : incrémental, ne revalide que les règles dont les dépendances ont changé (un edit splat n'invalide pas les règles mesh insert).
- Cache des résultats par règle, invalidé sélectivement.

### 4. « Aller à » navigation propre

Le bouton « Aller à » d'un problème :
- Déplace la caméra du viewport à la position du problème (avec une animation douce, pas un téléport brutal).
- Met le focus sur l'élément concerné dans l'Outliner si applicable.
- N'altère **jamais** l'état de l'éditeur (pas de changement d'outil, pas de modification de données).

### 5. Invariant terrain visible

Validation = lecture-seule sur les documents. Aucune mutation. Aucun render pass dédié. Animation caméra utilise la même infra que les autres mouvements caméra.

### 6. Une seule application

Pas de nouvel exécutable hors la suite `zone_validation_tests`.

### 7. Arbo server-core-style

- `src/world_editor/validation/` (nouveau).
- `src/world_editor/validation/rules/` (sous-dossier pour les règles individuelles).

## Portée

### Inclus

#### Partie A — Architecture générique

##### A.1 — `IValidationRule` interface

```cpp
namespace engine::editor::world::validation
{
    enum class Severity : uint8_t
    {
        Hint    = 0,
        Warning = 1,
        Error   = 2,
    };

    struct ValidationIssue
    {
        std::string ruleId;                  // "heightmap.holes"
        std::string title;                   // "Trou détecté dans la heightmap"
        std::string description;             // texte court, 1-2 phrases
        Severity severity;
        engine::math::Vec3 worldPosition;    // pour "Aller à"
        std::optional<engine::shared::Guid> targetEntityGuid;  // mesh insert, river, etc.
        std::string docSectionId;            // lien vers M100.47 si applicable
        std::string suggestedFix;            // texte court, optionnel
    };

    struct ValidationContext
    {
        const TerrainDocument*       terrain;
        const SplatDocument*         splat;
        const WaterDocument*         water;
        const MeshInsertDocument*    meshInserts;
        const InstanceDocument*      instances;
        // ... (read-only)
    };

    class IValidationRule
    {
    public:
        virtual ~IValidationRule() = default;
        virtual const char* GetRuleId() const = 0;
        virtual const char* GetCategory() const = 0;       // "heightmap", "water", ...
        virtual const char* GetDescription() const = 0;
        virtual Severity    GetDefaultSeverity() const = 0;

        /// Tags des documents que cette règle inspecte. Permet l'invalidation
        /// incrémentale : si un doc tagué change, la règle est re-runée.
        virtual std::vector<std::string> GetDocumentTags() const = 0;

        /// Exécute la validation, ajoute issues à la liste si nécessaire.
        virtual void Run(const ValidationContext& ctx,
                         std::vector<ValidationIssue>& issuesOut) const = 0;
    };
}
```

##### A.3 — `ZoneValidator` service principal

```cpp
class ZoneValidator
{
public:
    struct Options
    {
        bool                     onlyCategory;
        std::string              category;       // si onlyCategory
        std::set<std::string>    excludedRules;  // disable certaines règles
        Severity                 minSeverity = Severity::Hint;
    };

    struct Report
    {
        std::vector<ValidationIssue> issues;     // triés par severity desc
        double                       wallTimeMillis;
        uint32_t                     errorCount;
        uint32_t                     warningCount;
        uint32_t                     hintCount;
    };

    /// Exécute toutes les règles enregistrées et renvoie un rapport complet.
    /// Thread : peut être appelée depuis un worker (les documents sont
    /// thread-safe en lecture).
    Report Validate(const ValidationContext& ctx, const Options& options = {});

    /// Variante incrémentale : ne revalide que les règles touchant
    /// les documents `changedTags`.
    Report ValidateIncremental(const ValidationContext& ctx,
                                const std::vector<std::string>& changedTags,
                                const Report& previousReport);
};
```

- Sections collapsibles par sévérité.
- Filtre par catégorie (dropdown : tout, heightmap, water, mesh_inserts, etc.).
- Filtre par sévérité min.
- Recherche texte (titre, description).

- Lien 📖 Doc → ouvre F1 sur `docSectionId`.

##### C.3 — Validation avant export

Quand l'utilisateur déclenche `Fichier → Exporter la zone…` :

1. Lance une validation complète.
2. Si `errorCount > 0` : **modal bloquant** « Export impossible : 2 erreurs à corriger. [Voir les erreurs] [Annuler] ».
3. Si `warningCount > 0` et pas d'erreur : modal avec « 5 warnings détectés. Continuer l'export ? [Continuer] [Voir les warnings] [Annuler] ».
4. Si aucune issue : export direct.

##### C.4 — Indicateur live (background mode optionnel)

Si l'utilisateur active `Options → Validation continue` (off par défaut), une petite icône d'état dans la status bar :
- 🟢 OK (aucune issue)
- 🟡 N warnings/hints
- 🔴 N erreurs

Mise à jour incrémentale toutes les 5 secondes ou après chaque commande utilisateur (ce qui est plus rapide).

#### Partie D — Animation caméra « Aller à »

`CameraController::AnimateToPosition(target, durationMs = 800)` :
- Interpolation Bézier cubique entre position courante et target.
- Préserve l'angle de vue actuel (pas de tilt).
- Ajuste la distance pour que `target` soit centré dans le viewport.
- Pendant l'animation, les inputs caméra utilisateur sont ignorés.
- À la fin, déclenche un highlight visuel du target (ex : pulse 2 secondes autour de l'AABB du mesh insert concerné).

#### Partie E — Tests

`zone_validation_tests` :
- Round-trip de chaque règle : injection d'un cas test → issue produite.
- Pas de faux positif sur une zone vide saine.
- Performance : full pass sur zone 10×10 km avec 100 mesh inserts < 2 s.
- Incrémental : changement splat n'invalide pas les règles mesh insert.
- Ignore persiste : ignorer une issue → re-valider → l'issue n'apparaît plus.
- Modal bloquant : 1 error → bouton Exporter désactivé.
- Animation caméra : `AnimateTo` arrive à la position cible à ±0.1 m.

### Hors scope

- **Fix automatique généralisé** : seulement le bouton « Snap au sol » pour les mesh insert flottants est livré MVP. Les autres issues nécessitent l'intervention manuelle du mappeur.
- **Suggestions intelligentes** (« voici comment résoudre ce warning ») : le `suggestedFix` reste un texte court statique, pas un système expert. Évolution future avec M100.49 (Diagnostic).
- **Règles personnalisables par projet** : toutes les règles sont en dur dans le code et activables/désactivables, pas configurables (pas de DSL de validation).
- **Validation runtime in-game** : reste éditeur-only.
- **Validation cross-zones** (cohérence frontières inter-zones) : on valide une zone à la fois.
- **Partage de rapport** : pas d'export du rapport en PDF / JSON. Évolution future avec M100.51 (CI).
- **Localisation des messages d'erreur** : FR uniquement MVP.

  → Focus Outliner sur targetEntityGuid si applicable

  → ApplyMeshInsertReanchorCommand → CommandStack
  → Issue disparaît du report au prochain refresh

[Click "Ignorer"]
  → UserPrefs.ignoredIssueIds += issue.id
  → Persistance dans prefs.json
