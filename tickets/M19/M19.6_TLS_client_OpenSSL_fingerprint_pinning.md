# M19.6 — TLS client (OpenSSL) + certificate fingerprint pinning

## Objectif
Ajouter TLS côté client via OpenSSL et vérifier le serveur via fingerprint SHA-256 configurable (pinning).

## Dépendances
- M19.3
- M00.1

## Livrables
- `SSL_CTX` client + handshake TLS
- Extraction certificat serveur (X509)
- Calcul SHA-256 fingerprint (DER)
- Comparaison avec fingerprint config (normalisation)
- Refus explicite si mismatch

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Fingerprint: chaîne hex (avec/sans `:`) normalisée (case-insensitive, remove separators).
- Policy:
  - si fingerprint configuré → obligatoire
  - mode dev optionnel (allow_insecure_dev) uniquement via config explicite
- TLS min: 1.2+

## Étapes d’implémentation
1. Handshake TLS client (non-bloquant si thread réseau).
2. Récupérer cert serveur après handshake.
3. Calculer fingerprint SHA-256, normaliser, comparer.
4. Si mismatch: log + disconnect.
5. Si OK: read/write via SSL.

## Critères de validation (DoD)
- [ ] Fingerprint correct → connexion acceptée.
- [ ] Fingerprint incorrect → connexion refusée.
- [ ] Logs indiquent cause.

## Notes / pièges à éviter
- Ne pas “fallback” silencieusement en insecure.
