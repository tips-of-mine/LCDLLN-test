# M41.2_Item_duplication_prevention_+_transaction_logs

## Objectif
Prévenir duplication items via validation transactions, logs audit.

## Dépendances
- M16.3
- M35.3

## Livrables
- Atomic transactions (DB level)
- Validation: item exists, not equipped, owned by player
- Audit logs: all item transfers (trade, loot, mail, AH)
- Rollback tool (admin)

## Structure & chemins (verrouillé)
- Code moteur : uniquement sous `/engine/...`
- Contenu (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- Outils offline : uniquement sous `/tools/...`
- Tous les chemins de contenu sont relatifs à `paths.content` (config.json).
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

## Spécification technique
- Transactions: BEGIN -> validate -> execute -> COMMIT
- Validation:
  - Item exists in source inventory
  - Not equipped (can't trade equipped)
  - Owner matches requester
- Logs: item_transfer_log (timestamp, from, to, item, quantity, reason)

## Étapes d'implémentation
1. DB: item_transfer_log table
2. Trade transaction:
   ```sql
   BEGIN;
   -- Validate both players' items exist
   -- Remove items from player A
   -- Add items to player B
   -- Log transfer
   COMMIT;
   ```
3. On failure: ROLLBACK
4. Log every transfer (trade, loot, mail, vendor, drop)
5. Admin tool: query logs, detect suspicious patterns
6. Rollback: reverse transfer (emergency)

## Definition of Done (DoD)
- [ ] Build Windows OK (configure + build) via presets existants

- [ ] Aucun dossier racine non autorisé créé

## Notes / pièges à éviter
- Concurrency: lock inventory rows (SELECT FOR UPDATE)
- Retention: purge old logs (>6 months) pour DB size
