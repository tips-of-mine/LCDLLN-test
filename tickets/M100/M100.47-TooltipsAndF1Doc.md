## Résumé

Système d'aide intégré à l'éditeur, en deux volets :

1. **Tooltips riches** sur chaque slider, dropdown et bouton des `ToolPropertiesPanel`. Au survol prolongé (> 600 ms), un panneau s'affiche avec : nom du paramètre, description courte adaptée au mode (Simple = vulgarisée, Advanced = technique), illustration SVG inline montrant l'effet, plage de valeurs et défaut, et un lien « En savoir plus » qui ouvre la doc F1 sur la section correspondante.

2. **F1 = doc embarquée navigable**. La touche F1 (ou bouton ❓ dans chaque panel) ouvre une fenêtre docked avec une TOC à gauche, le contenu markdown rendu au centre, et une recherche plein-texte en haut. Le contenu vit dans `game/data/editor/docs/*.md`, en pur markdown éditable sans recompiler. MVP : 6 sections (Bienvenue, Outils terrain, Volumes 3D, Zone presets, Glossaire, FAQ).

Les tooltips et la doc partagent le même **store de contenu** (`HelpContentStore`) pour éviter la duplication : un paramètre comme `talusAngleDeg` a une définition unique consommée à la fois en tooltip et dans la doc.

Le ticket préserve l'invariant terrain visible et n'introduit aucun nouvel exécutable.

## Contexte critique à respecter

### 1. Contenu modifiable sans recompiler

Tous les textes (tooltips + doc) vivent dans `game/data/editor/`. Aucun string hardcodé dans le C++. Permet à un non-développeur de corriger une faute de frappe ou d'enrichir la doc via une PR markdown simple.

### 2. Tooltips s'adaptent au mode

En mode **Simple**, les tooltips affichent une explication vulgarisée (« Plus haut = vallées plus profondes »). En mode **Advanced**, ils affichent les détails techniques (« sedimentCapacity ∈ [1, 10], cf. Sebastian Lague hydraulic erosion model »). Le contenu est dans le même JSON, deux champs `description_simple` et `description_advanced`.

### 3. Illustrations SVG inline

Pour rester léger et offline, les illustrations sont des **SVG inline** dans le markdown. Pas de GIF (lourd), pas de vidéo (lourd + hors-scope). Une animation simple peut être faite via `<animate>` SVG natif. Pour les phénomènes complexes (érosion, watershed), un SVG static avec annotations suffit. Si besoin d'animation plus poussée, ticket art futur.

### 4. Aucune connexion réseau pour la doc

La doc fonctionne **100 % offline**. Les liens externes (GitHub, Discord) sont des références textuelles, pas des fetches en ligne.

### 5. Recherche plein-texte locale

Index construit au premier load, mis à jour si les markdown changent (hot-reload en debug). Algorithme : tokenize, stop-words FR/EN, ranking BM25 simplifié. Pas d'index persistant — reconstruction au boot (rapide, < 100 ms sur 6 sections).

### 6. Hooks tutoriel et diagnostic

`HelpSystem` expose des points d'extension pour M100.49 (Tutoriel) et M100.49 (Diagnostic). Ce ticket-ci les pose, M100.49 les remplit. Les hooks : `RegisterSection(string id)`, `JumpToSection(string id)`, `RegisterTutorialStep(...)`.

### 7. Invariant terrain visible

La fenêtre F1 est un panel ImGui standard, dockable, sans render pass dédié.

### 8. Arbo server-core-style

- `src/world_editor/help/` (nouveau).
- `game/data/editor/docs/*.md` (committés).
- `game/data/editor/tooltips/*.json` (committés).
- `game/data/editor/help_index/` (généré, .gitignored).

## Portée

### Inclus

#### Partie A — `HelpContentStore`

Service unique qui charge :

- Tous les `game/data/editor/docs/*.md` (markdown source).
- Tous les `game/data/editor/tooltips/*.json` (définitions de tooltips).
- Construit un index plein-texte au premier load.

```cpp
namespace engine::editor::world::help
{
    struct TooltipDefinition
    {
        std::string id;                       // "hydraulic_erosion.numDroplets"
        std::string label;                    // "Nombre de gouttes"
        std::string description_simple;       // "Plus = érosion plus marquée."
        std::string description_advanced;     // "Particles, typ. 50k-500k."
        std::string svgInline;                // SVG content, optionnel
        std::string defaultValue;             // "100000"
        std::string range;                    // "10000 - 500000"
        std::string docSectionId;             // "tool/hydraulic_erosion#numDroplets"
    };

    struct DocSection
    {
        std::string id;                       // "tool/hydraulic_erosion"
        std::string title;                    // "Érosion hydraulique"
        std::string markdownContent;          // brut
        std::vector<std::string> tags;        // ["erosion", "water", "tool"]
        std::vector<std::string> parentIds;   // hiérarchie TOC
    };

    class HelpContentStore
    {
    public:
        static HelpContentStore& Instance();

        void LoadFromContentPath();
        void ReloadHotswap();   // dev only

        const TooltipDefinition* FindTooltip(const std::string& id) const;
        const DocSection* FindSection(const std::string& id) const;
        std::vector<const DocSection*> Search(const std::string& query) const;
        std::vector<const DocSection*> GetTOC() const;  // racines de la hiérarchie

    private:
        std::unordered_map<std::string, TooltipDefinition> m_tooltips;
        std::unordered_map<std::string, DocSection> m_sections;
        SearchIndex m_index;
    };
}
```

#### Partie B — `RichTooltipWidget`


#### Partie E — Recherche

##### E.1 — Index BM25 simplifié

- Tokenize markdown content (sans markdown syntax).
- Stopwords FR + EN.
- Stemming léger (FR : verbe → racine, suffixes courants).
- Stockage : `unordered_map<string token, vector<{section_id, freq}>>`.
- Ranking : BM25 avec `k1 = 1.5`, `b = 0.75`.
- Snippet : 150 char autour du premier match, avec highlight.

##### E.2 — UI recherche

- Tape > 2 caractères → résultats live (debounce 200 ms).
- 10 premiers résultats affichés sous le champ.
- Clic → navigation vers la section, scroll automatique au match.

#### Partie F — Tests

- Tooltips : round-trip JSON, lookup par ID, fallback si introuvable.
- Markdown : parse correct, ancres extraites, hyperliens internes valides.
- Search : requêtes connues retournent les sections attendues.
- Hot-reload : modification d'un fichier MD pendant l'éditeur → contenu mis à jour sans crash.
- Bouton ❓ : ouvre la fenêtre F1 sur la bonne section.
- F1 : toggle on/off, dockable, historique nav.

## Schéma d'interaction utilisateur

```
[Survol prolongé d'un slider > 600 ms]
  → RichTooltipWidget détecte le hover
  → Lookup HelpContentStore.FindTooltip("hydraulic_erosion.numDroplets")
  → Si mode Simple → affiche description_simple + SVG inline
  → Si mode Advanced → affiche description_advanced + plage numérique
  → Bouton "📖 En savoir plus (F1)" en bas

[Click "📖 En savoir plus" OU touche F1]
  → HelpWindow s'ouvre
  → JumpToSection("tool/hydraulic_erosion#numDroplets")
  → TOC à gauche, contenu markdown rendu au centre
  → Scroll automatique à l'ancre #numDroplets

[Tape "érosion" dans la barre de recherche]
  → Debounce 200 ms
  → SearchIndex.Search("érosion") → ranking BM25
  → 10 résultats affichés avec snippets + highlight
  → Click sur un résultat → navigation vers section

[Click ❓ dans le header d'un ToolPropertiesPanel]
  → HelpWindow ouvre directement sur la section de l'outil
  → Ex: panel Hydraulic → ouvre "tool/hydraulic_erosion"

[Dev: Ctrl+Shift+H]
  → HelpContentStore.ReloadHotswap()
  → Re-parse tous les markdown + tooltips JSON
  → Si F1 ouverte, contenu mis à jour à chaud
```
