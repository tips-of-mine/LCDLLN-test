# M21.3 — Master applies pending migrations on startup

## Objectif
Le Master Server doit vérifier la version DB et appliquer automatiquement les migrations en attente au démarrage.

## Dépendances
- M19.4 (server)
- M21.2

## Livrables
- `MigrationRunner`:
  - connect DB
  - read current version
  - list pending migrations
  - apply in order
  - update schema_version
- Logs détaillés

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Locking: empêcher deux masters d’appliquer en même temps (advisory lock MySQL).
- Stop-on-failure: pas de boot partiel.

## Étapes d’implémentation
1. Implémenter advisory lock.
2. Lire schema_version.
3. Appliquer migrations.
4. Mettre à jour schema_version.

## Critères de validation (DoD)
- [ ] Migrations s’appliquent automatiquement.
- [ ] Concurrence master protégée.
