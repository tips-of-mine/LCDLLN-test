# M35.3_Player_trading_direct_trade_window_+_security

## Objectif
Échange direct player-to-player avec fenêtre trade, confirmation, anti-scam.

## Dépendances
- M35.1
- M16.3

## Livrables
- Trade request/accept
- Trade window UI (both players items + gold)
- Lock & confirm mechanism
- Trade logging (audit)

## Structure & chemins (verrouillé)
- Code moteur : uniquement sous `/engine/...`
- Contenu (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- Outils offline : uniquement sous `/tools/...`
- Tous les chemins de contenu sont relatifs à `paths.content` (config.json).
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

## Spécification technique
- Trade flow:
  1. Player A requests trade
  2. Player B accepts
  3. Both add items/gold
  4. Both lock (ready)
  5. Both confirm (final)
  6. Server validates, swaps items atomically
- Lock: prevent adding more items (review phase)
- Confirm: final accept (irreversible)

## Étapes d'implémentation
1. /trade <name>: send TradeRequest
2. Accept: create Trade session (both players)
3. UI: split window (left=self, right=other)
4. Add items: drag&drop, update Trade session
5. Lock button: set ready flag
6. Both locked: enable Confirm button
7. Both confirm: server validates (items exist, not equipped), swap inventories
8. Log trade (player_trade_log)
9. On cancel: return items

## Definition of Done (DoD)
- [ ] Build Windows OK (configure + build) via presets existants
- [ ] Exécutable lancé (smoke test) et pas de crash immédiat
- [ ] Aucun dossier racine non autorisé créé
- [ ] Rapport final: fichiers modifiés + commandes + résultats + DoD

## Notes / pièges à éviter
- Scam prevention: show items clearly, 5s review after lock
- Range check: both players <10m
