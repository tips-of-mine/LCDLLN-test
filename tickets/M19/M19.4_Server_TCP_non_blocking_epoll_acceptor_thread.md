# M19.4 — Server TCP non-blocking (Linux epoll) + acceptor thread

## Objectif
Créer le cœur serveur TCP non-bloquant (Linux) basé sur epoll, avec thread d’accept dédié et pool de workers configurable.

## Dépendances
- M19.2
- M00.1
- M00.3

## Livrables
- `NetServer`:
  - acceptor thread
  - epoll loop IO
  - objets connexion (fd, buffers, state)
  - RX framing + TX flush
- Worker pool pour traitement packets (sans bloquer IO)
- Backpressure (cap TX queue)

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- epoll sans busy-loop (edge/level au choix, mais stable).
- RX read-until-EAGAIN, parse frames via header `size`.
- TX queue + EPOLLOUT uniquement si queue non vide.
- Limits:
  - max queued TX bytes/connection (disconnect si dépassement)
  - max connections (config)

## Étapes d’implémentation
1. Acceptor: accept, set nonblocking, register epoll.
2. epoll loop: handle IN/OUT/ERR/HUP.
3. RX: read, append, parse frames, push jobs vers workers.
4. TX: flush queue jusqu’à EAGAIN.
5. Cleanup: close fd, libérer buffers, log reason.

## Critères de validation (DoD)
- [ ] 1000 connexions locales stables (outil simple).
- [ ] CPU idle correct sans trafic.
- [ ] Pas de crash sur disconnect brutaux.

## Notes / pièges à éviter
- Traiter EPOLLERR/HUP systématiquement.
- Ne jamais exécuter DB/game logic dans IO thread.
