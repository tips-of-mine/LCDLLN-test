# M22.4 — Ticket system: client connects to shard only with Master-issued ticket

## Objectif
Empêcher toute connexion directe à un shard sans ticket temporaire émis par le Master.

## Dépendances
- M20 (sessions)
- M22.1 (registry)

## Livrables
- Ticket struct:
  - ticket_id (128-bit recommandé)
  - account_id
  - target_shard_id
  - expires_at (court, ex 30–60s)
  - signature/HMAC (v1 simple)
- Opcodes:
  - REQUEST_SHARD_TICKET
  - SHARD_TICKET_RESPONSE
- Validation ticket côté shard

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `textures/brick_albedo.png` depuis `${paths.content}/textures/brick_albedo.png`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.


## Spécification technique
- Ticket à usage unique (consume on first use).
- Expire rapidement.
- v1: HMAC avec secret shard/master partagé (ou secret master + distribution).

## Étapes d’implémentation
1. Définir payloads.
2. Implémenter création ticket côté master.
3. Implémenter validation + consume côté shard.
4. Tests: ticket expiré/refusé, ticket valide accepté.

## Critères de validation (DoD)
- [ ] Un shard refuse sans ticket.
- [ ] Ticket valide ouvre session shard.
