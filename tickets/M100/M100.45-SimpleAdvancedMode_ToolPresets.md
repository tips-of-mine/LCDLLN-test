# M100.45 — Simple/Advanced Mode Toggle + Tool Parameter Presets (Phase 12 — Accessibilité)

## Résumé

**Démarre la Phase 12 « Accessibilité éditeur »** en livrant deux mécaniques transverses qui s'appliquent à **tous les outils** déjà existants (M100.6, .7, .9, .13, .17, .35-43) et futurs :

1. **Mode Simple/Advanced** : un toggle global dans `Options` qui bascule l'éditeur entre deux niveaux d'exposition de paramètres. En mode **Simple** (par défaut au premier lancement), chaque `ToolPropertiesPanel` n'affiche que 3-4 sliders essentiels par outil, masquant tout le reste. En mode **Advanced**, l'intégralité des paramètres est exposée comme aujourd'hui. La préférence est persistée par utilisateur (`game/data/editor/user_prefs.json`).

2. **Tool Parameter Presets** : par outil, un catalogue de 3-5 presets nommés (ex : Hydraulic Erosion → « Légère / Réaliste / Intense / Désertique »). Un dropdown en haut de chaque `ToolPropertiesPanel` applique le preset sélectionné aux sliders. Les presets sont définis en JSON sous `game/data/editor/tool_presets/<tool_id>.json`, modifiables sans recompiler.

Ces deux mécaniques fonctionnent **ensemble** : en mode Simple, le dropdown de presets est mis en avant comme l'interaction primaire (« choisis ton type d'érosion »), et les 3-4 sliders restants permettent un fine-tuning léger. En mode Advanced, le dropdown reste accessible mais tous les paramètres sont éditables.

**Aucun nouvel exécutable.** Code dans `engine_core`, binaire éditeur unique.

## Dépendances

- **M100.1** — World Editor Bootstrap (config loading).
- **M100.35** — Toolbar à icônes (le panneau d'outils existe déjà).
- Tous les outils déjà livrés : M100.6, .7, .9, .13, .17, .35-43.

Pas de dépendance bloquante sur Phase 13 (décoration) ou tickets futurs de Phase 12.

## Contexte critique à respecter

### 1. Migration outil par outil, pas big-bang

Ce ticket touche **13 outils existants** (Sculpt, Stamp, Splat, Lake, River, Coastline, Mountain/Valley, River Network, Hydraulic, Thermal/Wind, Cave, Overhang, Arch, Dungeon). Plutôt qu'une migration big-bang risquée, le ticket procède en **2 phases internes** :

- **Phase A** : Infrastructure transverse (toggle, registry, prefs, base classe). Aucun outil modifié. Aucun changement utilisateur visible.
- **Phase B** : Migration outil par outil. Chaque outil reçoit sa définition de mode Simple (quels sliders garder visibles) + son fichier `tool_presets/<id>.json`. Tests par outil pour vérifier qu'aucune fonctionnalité Advanced n'a régressé.

Cette structure permet à Codex de livrer la phase A en premier, validation, puis B en sous-PRs par groupe d'outils.

### 2. Backward compat des outils non-migrés

Pendant la phase B, certains outils peuvent ne pas encore être migrés. Ils doivent **continuer à fonctionner** comme aujourd'hui en mode Advanced (mode par défaut pour eux tant que la migration n'est pas faite), avec un flag `IsModeAwareInPropertiesPanel() = false` qui dit au système « cet outil ne respecte pas encore Simple/Advanced ».

### 3. Préférences persistées par utilisateur

`game/data/editor/user_prefs.json` est nouveau. Il vit dans le content path mais **n'est pas committé** dans git (ajout au `.gitignore`). Schéma initial :

```json
{
  "version": 1,
  "editorMode": "Simple",
  "lastPresetByTool": {
    "hydraulic_erosion": "realistic",
    "mountain_macro": "alpine_range"
  },
  "showAdvancedTooltips": false,
  "tutorialCompletionFlags": {}
}
```

Le ticket M100.49 (Tutoriel) consommera `tutorialCompletionFlags`. Le ticket M100.47 (Tooltips) consommera `showAdvancedTooltips`. Ce ticket-ci ajoute juste le squelette.

### 4. Pas de toolbar duplication en mode Simple

La toolbar à icônes (M100.35) reste **identique** en mode Simple et Advanced. Les outils ne disparaissent pas. C'est le `ToolPropertiesPanel` qui se simplifie. Un utilisateur peut donc en mode Simple cliquer sur Hydraulic Erosion et voir 3 sliders au lieu de 9 — mais l'outil reste accessible.

### 5. Invariant terrain visible

Inchangé. Le toggle de mode ne touche rien au rendu.

### 6. Une seule application

Pas de nouvel exécutable hors la suite `editor_modes_tests`.

### 7. Arbo server-core-style

- `src/world_editor/modes/` (nouveau dossier).
- `src/world_editor/presets/` (nouveau dossier).
- `game/data/editor/user_prefs.json` (à .gitignore).
- `game/data/editor/tool_presets/*.json` (committés).

#### Phase A — Infrastructure transverse

##### A.1 — `EditorModeRegistry`

```cpp
namespace engine::editor::world::modes
{
    enum class EditorMode : uint8_t
    {
        Simple   = 0,
        Advanced = 1,
    };

    /// Centralise l'état du mode courant. Singleton accessible depuis tous
    /// les panels. Persisté dans user_prefs.json.
    class EditorModeRegistry
    {
    public:
        static EditorModeRegistry& Instance();

        EditorMode GetCurrentMode() const;
        void       SetCurrentMode(EditorMode mode);  // persists immediatement

        /// Subscribe pour être notifié quand le mode change.
        /// Le callback est invoqué synchrone depuis SetCurrentMode().
        using ChangeCallback = std::function<void(EditorMode)>;
        size_t Subscribe(ChangeCallback cb);
        void   Unsubscribe(size_t handle);

    private:
        EditorMode m_mode = EditorMode::Simple;
        std::unordered_map<size_t, ChangeCallback> m_callbacks;
        size_t m_nextHandle = 0;
    };
}
```

##### A.2 — `IToolPropertiesPanelContent` interface

Chaque outil expose son panneau via une interface qui sait quels paramètres montrer selon le mode :

```cpp
namespace engine::editor::world::ui
{
    class IToolPropertiesPanelContent
    {
    public:
        virtual ~IToolPropertiesPanelContent() = default;

        /// Le titre du panel (ex: "Hydraulic Erosion").
        virtual const char* GetDisplayName() const = 0;

        /// L'identifiant stable pour les presets (ex: "hydraulic_erosion").
        virtual const char* GetToolId() const = 0;

        /// True si l'outil sait gérer Simple/Advanced. Pendant la phase de
        /// migration, certains outils retournent false et restent toujours
        /// en mode Advanced.
        virtual bool IsModeAwareInPropertiesPanel() const = 0;

        /// Render le contenu. La modes::EditorMode courante est passée pour
        /// que l'outil sache quels paramètres masquer.
        virtual void RenderPropertiesPanel(modes::EditorMode mode) = 0;

        /// Renvoie la liste des presets disponibles (chargée depuis
        /// tool_presets/<toolId>.json). Vide si pas de presets.
        virtual const std::vector<presets::ToolPreset>& GetAvailablePresets() const = 0;

        /// Applique un preset aux paramètres internes de l'outil.
        virtual void ApplyPreset(const presets::ToolPreset& preset) = 0;
    };
}
```

Les outils existants (`HydraulicErosionTool`, etc.) implémentent cette interface (ou exposent un sous-objet qui l'implémente).

## Spécification fonctionnelle

### Au premier lancement

1. Pas de `user_prefs.json` → l'éditeur le crée avec les défauts (`editorMode = Simple`).
2. Tous les `ToolPropertiesPanel` apparaissent en mode Simple.
3. Un toast informatif s'affiche au premier clic sur un outil : « Mode Simple actif. Pour accéder à tous les paramètres : Options → Mode éditeur → Avancé ». Toast dismissible.

### Workflow user typique

```
[ouvre l'éditeur]
  → toolbar visible (toutes les icônes)
  → ToolPropertiesPanel vide ou avec dernier outil

[click Hydraulic Erosion]
  → panel s'affiche :
    [Preset: Réaliste ▼]  [ⓘ]
    Intensité globale     |■■■■□□□□□□|
    Nombre de gouttes     |■■■■■□□□□□|
    Niveau de détail      |■■■■■■□□□□|
    [▶ Simulate]

[click sur Preset dropdown → "Intense"]
  → tous les sliders se mettent aux valeurs du preset
  → user_prefs.json mis à jour : lastPresetByTool["hydraulic_erosion"] = "intense"

[tweak un slider après le preset]
  → dropdown passe à "Custom (modifié)"

[Options → Mode éditeur → Avancé]
  → panel se re-render avec TOUS les sliders
  → user_prefs.json mis à jour : editorMode = Advanced
```

### Validation des presets

À l'application d'un preset, valider que les `parameters` du JSON correspondent au schéma attendu de l'outil. Si un champ manque ou est invalide : log warning, ne pas crash, laisser les valeurs actuelles inchangées pour les champs incorrects.
