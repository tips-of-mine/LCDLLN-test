# M33.1_Auth_registration_+_login_+_session

## Objectif
Système authentification sécurisé avec enregistrement, login, session tokens.

## Dépendances
- M13.1

## Livrables
- Auth server (separate ou intégré)
- Registration: username, email, password
- Login: validate credentials, issue token
- Session management: JWT ou session ID
- Password hashing (bcrypt/argon2)
- Schéma DB MVP: accounts, servers, races, characters (voir Spécification)

## Structure & chemins (verrouillé)
- Code moteur : uniquement sous `/engine/...`
- Contenu (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- Outils offline : uniquement sous `/tools/...`
- Tous les chemins de contenu sont relatifs à `paths.content` (config.json).
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

## Spécification technique
- Password: bcrypt (cost 12) ou argon2
- Session token: JWT ou random UUID + DB
- Token expiry: 24h, refresh token 30d
- HTTPS obligatoire (production)
- Rate limiting: 5 login attempts / 5min
- Schéma DB MVP:
  - accounts(id, username, email, password_hash, created_at, modified_at, active_session)
  - servers(id, name, region, status)
  - races(id, name, description)
  - characters(id, account_id, server_id, race_id, class_id, name, level, appearance_json, created_at, modified_at)
  - Contraintes: username/email uniques, name unique par server, max 5 characters par account/server, FK account/server/race/class
  - Suppression personnage par utilisateur (libère un slot), transfert inter-serveur réservé admin
  - servers peuvent être initialisés via config pour MVP, mais schéma DB prévu
  - races data-driven (pas hardcodé), classes peuvent être filtrées par race

## Étapes d'implémentation
1. DB: accounts (id, username, email, password_hash, created_at, modified_at, active_session)
2. DB: servers (id, name, region, status) et races (id, name, description)
3. DB: characters (account_id, server_id, race_id, class_id, appearance_json, name, level, timestamps)
4. /register: validate input (email format, password strength)
5. Hash password (bcrypt), insert DB
6. /login: fetch account, verify password hash
7. Generate session token (JWT: {userId, exp})
8. Client: store token (secure storage)
9. Game server: validate token on connect
10. Refresh token flow (optional MVP)

## Definition of Done (DoD)
- [ ] Build Windows OK (configure + build) via presets existants
- [ ] Exécutable lancé (smoke test) et pas de crash immédiat
- [ ] Aucun dossier racine non autorisé créé
- [ ] Rapport final: fichiers modifiés + commandes + résultats + DoD

## Notes / pièges à éviter
- NEVER log passwords
- Email verification (send code) phase 2
- 2FA optionnel phase 3
