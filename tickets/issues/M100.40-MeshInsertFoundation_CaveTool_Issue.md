# M100.40 — Mesh Insert Foundation + Cave Tool (Phase 11 — Volumes 3D)

## Résumé

**Démarre la Phase 11 « Volumes 3D »** en livrant deux couches superposées :

1. **`MeshInsertSystem` — la fondation partagée** : un système de placement de meshes glTF arbitraires « par-dessus » la heightmap, avec collision proxy, lighting probe locale, audio reverb optionnel, gestion du streaming, persistance binaire dans `instances/mesh_inserts.bin` (nouveau format), édition gizmo (translate/rotate/scale), undo/redo, et hooks vers `SurfaceQueryService` (M100.11) pour le gameplay (« je suis dans un volume 3D nommé »). Cette fondation sert aussi M100.41 (overhangs), M100.42 (arches), M100.43 (donjons).

2. **`CaveTool` — outil spécialisé grottes** : UI dédiée pour poser des entrées de grotte sur le terrain. Catalogue d'assets glTF de grottes (`game/data/meshes/caves/*.gltf`), camouflage automatique de l'entrée par un patch de splat « rocher » + props (gros rochers cachant les bords), portail logique pour transitions extérieur ↔ intérieur, anchoring sur la heightmap (rotation automatique pour aligner l'orientation à la pente locale).

Ces deux livrables sont dans **un seul ticket / une seule PR** car la fondation seule serait inutilisable sans au moins un outil consommateur, et `CaveTool` est le scénario qui dimensionne le mieux les besoins (volume intérieur, entrée camouflée, contraintes navmesh).

**Aucun nouvel exécutable n'est créé.** Code dans `engine_core`, binaire éditeur unique.

## Contexte critique à respecter

### 1. Heightmap ≠ supprimée sous le mesh insert

Le mesh insert **ne perce pas** la heightmap. Le terrain sous-jacent reste à sa hauteur originale ; le mesh glTF est dessiné par-dessus, et son intérieur est accessible via une **« porte logique »** (un volume trigger à l'entrée). Le rendu du terrain dans le sous-sol de la grotte (sous la heightmap apparente) est délibérément absent — le joueur ne le voit pas car il est à l'intérieur du mesh.

Conséquence importante : **pas besoin de modifier `terrain.bin`**. Pas de carving. Le mesh insert vit en parallèle.

### 2. Pas de génération de caves procédurales dans ce ticket

L'utilisateur **place des meshes pré-faits** depuis un catalogue. La génération procédurale de meshes de grottes (Voronoi 3D, marching cubes, etc.) est explicitement hors scope. Tous les meshes sont des **assets glTF authorés** sous `game/data/meshes/caves/`.

### 3. Cohabitation avec la phase 2.5

- Les outils érosion (M100.38, M100.39) **ne modifient pas** les mesh inserts. Si l'utilisateur érode autour d'une grotte existante, le mesh reste en place ; seul le terrain alentour change. L'anchoring de la grotte (point de contact entrée ↔ terrain) peut devenir incohérent si l'érosion a creusé une vallée sous l'entrée — un warning est affiché dans le Inspector si la distance verticale terrain ↔ entrée dépasse 2 m.

### 4. Bridge vmap — préparation finale

Les mesh inserts ont des proxies collision qui devront être propagés au serveur (`src/shardd/internals/vmap/`). Le **bridge final** est rédigé en M100.44. Ce ticket M100.40 prépare le terrain en :
- nommant ses formats binaires de façon symétrique au server (`mesh_inserts.bin` + `mesh_inserts_collision.bin`),
- exportant un `collision_proxy_index.json` listant les meshes à intégrer aux tiles vmap au prochain bake,
- documentant les contraintes que M100.44 implémentera.

### 5. Invariant terrain visible (rappel)

Le mesh insert est un objet 3D rendu via `MeshAsset` standard (réutilise la pipeline existante M03+). Pas de nouveau render pass. Pas de touche au frustum cull / camera / fallback orange.

### 6. Une seule application

Pas de nouvel exécutable hors la suite `mesh_insert_tests` + `cave_tool_tests`.

### 7. Arbo server-core-style

- `src/world_editor/volumes/` (nouveau dossier dédié Phase 11).
- `src/world_editor/volumes/caves/` pour les éléments spécialisés grottes.
- `src/client/world/volumes/` pour le rendu / streaming runtime.
- `game/data/meshes/caves/` pour les assets glTF.

## Portée

### Inclus

#### Partie A — `MeshInsertSystem` fondation

##### A.1 — Données

```cpp
struct MeshInsertInstance
{
    engine::shared::Guid guid;
    std::string          gltfRelativePath;  // ex: "meshes/caves/cave_small_01.gltf"
    engine::math::Vec3   worldPosition;     // point d'ancrage (typ. centre porte)
    engine::math::Vec3   eulerRotationDeg;  // rotation XYZ degrés (Y dominant)
    float                uniformScale = 1.0f;
    std::string          insertCategory;    // "cave", "overhang", "arch", "dungeon"
    std::string          displayName;       // libre, pour Outliner
    bool                 hasInteriorVolume = true;
    bool                 castsShadow       = true;
    bool                 receivesAudioReverb = true;
    float                lightProbeIntensity = 1.0f;
    bool                 allowsWaterIngress  = false;  // si vrai, l'eau M100.13 peut entrer
};
```

##### A.2 — Format binaire

- Fichier : `instances/mesh_inserts.bin`.
- Magic : `LCMI` (Lcdlln Mesh Inserts).
- Version : `kMeshInsertsBinVersion = 1`.
- Header : nombre d'instances + offset table.
- Chaque instance : champs du struct sérialisés en little-endian, strings length-prefixed UTF-8.
- Fichier compagnon : `instances/mesh_inserts_collision.bin` (table de hash → AABB monde pour query rapide, bake offline lors de l'export).

##### A.3 — Document éditeur

`MeshInsertDocument` (`src/world_editor/volumes/MeshInsertDocument.h/.cpp`) :
- Possède le `std::vector<MeshInsertInstance>` édité.
- Sait charger/sauvegarder le binaire.
- Méthodes : `Add(MeshInsertInstance)`, `Remove(Guid)`, `Update(Guid, MeshInsertInstance)`, `GetByGuid(Guid)`, `GetByCategory(string)`.
- Émet des events `OnInstanceAdded`, `OnInstanceRemoved`, `OnInstanceUpdated` consommés par les panels.

##### A.4 — Rendu

- Réutilise `MeshAsset` (M03.x) pour charger les glTF.
- Cache de glTF partagé : `MeshInsertCache` qui charge chaque `gltfRelativePath` une seule fois et la partage entre toutes les instances.
- Pipeline rendu : pas de nouveau pass, ajout des instances à la passe `Geometry` standard (forward LOD + deferred opaque).
- Lighting probe locale optionnelle : si `lightProbeIntensity > 0`, une probe IBL est générée à `worldPosition + (0, 2, 0)` (2 m au-dessus du sol intérieur) lors du bake. Bake offline, pas runtime.
- Collision proxy : chaque mesh insert charge son `.collision.bin` (M100.12 format) lors du streaming et l'enregistre auprès du `CollisionSystem`.

##### A.5 — Streaming

- Les mesh inserts sont indexés par `GlobalChunkCoord` à l'export (en fonction de `worldPosition`).
- Le `StreamCache` charge le `MeshAsset` + `.collision.bin` quand le chunk hébergeant l'instance entre dans le ring `Visible` ou `Active`.
- Décharge quand le chunk sort du ring `Loaded`.

##### A.6 — Audio reverb hook

- Si `receivesAudioReverb = true`, le mesh insert pose un volume de reverb à son AABB (qui doit englober l'intérieur jouable). Le volume est géré par M100.X audio (référence : `engine::audio::ReverbZoneManager` existant ou à câbler).
- Ce ticket fournit le **flag** et la métadonnée AABB ; l'intégration runtime peut être complétée dans un follow-up audio si besoin (hors scope ici).

##### A.7 — Édition gizmo

- L'outil parent (`CaveTool`, futur `OverhangTool`, etc.) délègue le gizmo au `EasyPlacementTool` (M100.17).
- Gizmo standard translate / rotate (Y dominant) / scale uniforme.
- Snap au sol (raycast vers `Terrain` pour aligner `worldPosition.y`).
- Snap rotation à 15° pas pour la stabilité visuelle.

##### A.8 — Surface query intégration

- Quand le joueur entre dans le volume AABB d'une instance avec `hasInteriorVolume = true`, le `SurfaceQueryService` (M100.11) reçoit un override : `currentVolumeId = <guid>`, `currentVolumeCategory = "cave"`.
- Cela permet aux systèmes gameplay (chat « region », audio ambiance, météo bloquée, etc.) de réagir.

##### B.3 — Workflow

1. Cliquer **Cave** dans la toolbar.
2. Sélectionner un mesh dans le catalogue.
3. Cliquer sur le terrain → placement preview (mesh fantôme semi-transparent).
4. Tweak rotation / scale / camouflage en live.
5. **Place** → `PlaceCaveCommand` poussée :
   - Crée la `MeshInsertInstance` dans `MeshInsertDocument`.
   - Si camouflage splat coché : écrit le patch splat « Rocher » via `SplatDocument` (delta sparse).
   - Si props auto coché : crée N instances de props rochers via `InstanceDocument` (M100.17 format).
6. **Cancel** ou changer d'outil → abandon de la preview.

##### B.4 — Commit

- `PlaceCaveCommand` regroupe les 3 sub-commands logiques (mesh insert + splat patch + props auto) en une seule unité réversible.
- Undo retire la grotte + restore le splat + retire les props.

```
[click sur terrain]
  → place preview confirmé, l'instance n'est PAS encore committée
  → bouton "Place" devient actif

[click Place]
  → PlaceCaveCommand pushée :
      MeshInsertDocument.Add(instance)
      SplatDocument.ApplyDelta(camouflage)
      InstanceDocument.AddProps(rocks)
  → LOD regen pour les chunks touchés par le splat
  → MeshInsertCache charge le glTF si pas déjà fait
  → l'outil reste actif (prêt à placer une autre grotte)

[click Cancel]  → preview annulée, document inchangé

[click autre bouton toolbar]  → confirmation modale si preview en cours

[Ctrl+Z]  → undo : retire la grotte + restore splat + retire props
```

### Snap au sol

Au moment du `Place`, la `worldPosition.y` du mesh est ajustée pour que `worldPosition.y = TerrainChunk::SampleHeight(x, z) - entrancePoint.y` où `entrancePoint` vient du catalog. Ainsi le point d'entrée du mesh coïncide exactement avec le sol.

### Auto-rotation pour pente

Optionnel via une checkbox (par défaut OFF) : la rotation X et Z est ajustée pour aligner l'orientation à la normale du terrain au point d'entrée (lissée sur un voisinage 5×5 cells). Limité à ±20° pour éviter des grottes complètement penchées.
