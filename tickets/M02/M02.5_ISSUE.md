# M02.5 — Asset system: mesh + texture loading

## Objectif
Créer un système d'assets MVP (handles, cache, cycle de vie) pour charger meshes et textures utilisés par le renderer.

## Dépendances
- M02.4

## Livrables
- Asset registry + handles stables
- Loader mesh (CPU -> GPU)
- Loader textures (sRGB/linear)

## Structure & chemins (verrouillé)
- **Code moteur** : uniquement sous `/engine/...`
- **Contenu** (textures/meshes/json/audio) : uniquement sous `/game/data/...`
- **Outils offline** : uniquement sous `/tools/...`
- Tous les chemins de contenu sont **relatifs** à `paths.content` (config.json).  
  Exemple : charger `meshes/crate.glb` depuis `${paths.content}/meshes/crate.glb`.
- ❌ Interdit : créer un dossier racine `/assets` ou utiliser des chemins absolus.

## Spécification technique
- API d'accès: `AssetHandle<T>` + lookup par path.
- Référencement compté ou ownership explicite pour éviter use-after-free.
- Cache: éviter rechargement d'un même fichier déjà en mémoire.

## Étapes d'implémentation
1. Définir `AssetId`, `AssetHandle`, `AssetRegistry`.
2. Implémenter loader mesh minimal + upload vertex/index buffers.
3. Implémenter loader textures + conversion flags sRGB/linear.
4. Ajouter API de demande asynchrone/synchrone selon besoin MVP.

## Critères de validation (DoD)
- [ ] Deux demandes du même asset partagent la même ressource.
- [ ] Mesh + texture de test se chargent et sont rendus.
- [ ] Les ressources sont libérées proprement à la fermeture.

## Notes / pièges à éviter
- Clarifier dans M11.1 la stratégie glTF (sans nouvelle dépendance externe).
